## Data layer

### Schema overview (12 tables)

```
families ─── members ─── sessions
              │  │
              │  └─── otp_codes (by phone, no FK)
              │
              ├─── quiz_results (member optional → anonymous via session_id)
              ├─── purchases
              ├─── documents             (Block 3)
              ├─── notifications
              └─── invitations (inviter + invitee)

subscriptions ─── families  (1:1)
consultations ─── families  (member optional → anonymous form)
```

All ids are **cuid2** strings (24 char), except `sessions.id` which is the
session token itself (varchar 64). All timestamps are `timestamptz` with
`now()` default.

### DAL contract

Every domain table has a query module under `lib/db/queries/`. Convention:

```ts
import { eq } from 'drizzle-orm'
import { db } from '@/lib/db'
import { members, type Member, type NewMember } from '@/lib/db/schema'

export async function getMemberByPhone(phone: string): Promise<Member | null> {
  const rows = await db.select().from(members).where(eq(members.phone, phone)).limit(1)
  return rows[0] ?? null
}
```

Rules:
- **No raw SQL** in app code. If you need it, escape via Drizzle's
  `sql` helper exported from `@/lib/db/schema`.
- **No `as any`**. The schema's `$inferSelect` / `$inferInsert` give you
  exact row types. Use them.
- **Never call `db.*` from a Client Component**. Only Route Handlers,
  Server Components, and Server Actions.

### Adding a new table

1. Edit `lib/db/schema.ts`. Add the `pgTable` definition + types.
2. `npm run db:generate` — drizzle-kit writes a SQL migration into
   `lib/db/migrations/`.
3. Review the SQL. Check column nullability + indexes.
4. Add a query module under `lib/db/queries/`.
5. `npm run db:migrate` runs migrations against `DATABASE_URL`.
6. Bump `docs/architecture.md` if the relationship is novel.

### Why DAL not direct queries in routes

- Routes stay thin and easy to test.
- Query reuse across routes / cron / scripts.
- Easy to swap underlying engine (e.g. add caching, batching).

## Auth flow

```
POST /api/auth/send-otp    { phone }
  → otpCodes.createOtpCode(phone)
  → returns { ok: true } (code is logged in dev, sent via SMS in Block 2)

POST /api/auth/verify-otp  { phone, code }
  → otpCodes.consumeOtpCode(phone, code)
  → setUserSession(phone) → members.getOrCreateByPhone → sessions.create
  → sets httpOnly cookie 'tebiq_user_session' (30d)

GET  /api/auth/me
  → getCurrentUser() reads cookie → sessions.getValid → members.getById

POST /api/auth/logout
  → clearUserSession() destroys session row + clears cookie
```

`getCurrentUser()` returns `Member | null`. Older code typed this as
`User`; the alias `import type { User } from '@/lib/auth/session'` still
works.

## API response envelope

`lib/api/response.ts` exports `ok()` / `err()` helpers. New routes use
them:

```ts
import { ok, err, errors } from '@/lib/api/response'

export async function GET() {
  const user = await getCurrentUser()
  if (!user) return errors.unauthorized()
  const data = await listPurchasesByFamilyId(user.familyId)
  return ok({ purchases: data })
}
```

Existing routes that have UI consumers (consultation form, admin panel)
still return the legacy un-enveloped shape. Block 2 will migrate them in
lockstep with consumer updates.

## Static data validation

All `code-as-data` modules under `lib/knowledge/` and `lib/check/`
declare a zod schema and validate the constant array at module load:

```ts
const _data: Concept[] = [ /* ... */ ]
export const CONCEPTS = z.array(ConceptSchema).parse(_data)
```

This catches typos in the data (missing field, bad enum) at app startup
rather than at first read. Tradeoff: a bad commit fails the build.

## Cron security

`/api/cron/check-immigration` requires
`Authorization: Bearer ${CRON_SECRET}`. Vercel Cron sets this header
automatically when the env var is configured. Without `CRON_SECRET`
set, the route allows unauthenticated calls (dev mode); production
deploys MUST set it.

## What's still on KV (intentional)

These are operational caches that don't need transactional storage:

- `monitor:{id}` — last SHA-1 hash + lastChecked (cron rewrites daily)
- `monitor:alert:{id}` — change-detected flag
- `processing-time:{visa}` — scraped审查処理期間 cache
- `stats:*` — public counters (homepage, admin dashboard)
- `admin:error_log` — rolling 100-entry list
- `healthcheck:ping` — system-status probe
- `share:{id}` — 7-day TTL share-link payload (tiny, ephemeral)
- `reminder:{phone}` — opt-in reminder config (small key-value)

These can migrate to Postgres when there's a reason (reporting,
multi-region read consistency). Not before.

## Tests

Unit tests live next to the module under test, suffixed `.test.ts`.
Run with `npm test` (uses `tsx --test`, no Jest dependency).

Database tests **mock the entire `@/lib/db` module** so they run without
a live Postgres. Integration tests against a real DB land in Block 2.
