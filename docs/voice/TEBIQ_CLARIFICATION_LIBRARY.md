# TEBIQ Clarification Message Library

> 本文件是 TEBIQ 向用户请求补充信息时的澄清消息库。
>
> 当管线判断当前可用信息不足以可靠地完成路由或风险检查，进入 `clarification_needed` 状态时，使用本库中的模板生成引导文字。
>
> 所有 `user_copy` 均为 B 层（待验证），适用于 Preview 和 Internal Console 预览状态，不用于当前生产用户端。

---

## 使用原则

1. 每次只问**一个问题**。不允许把多个 clarification 堆在一起问用户。
2. 语气：**直接、简短**。不解释为什么要问，不道歉，不用"打扰一下"之类的开场。
3. 选项：如果可以给选项，给选项。不强迫用户用开放文本回答可以用选择题解决的问题。
4. 不要让用户感到被质疑。措辞中性，不暗示用户填错了或信息不完整。
5. 不要问出内部字段名。不用"`domain`"、"`intent`"、"`在留类型字段`"等系统术语。

## 澄清模板

### CL-01 — 当前在留资格类型不明

```yaml
clarification_code: CL-01
trigger_condition: |
  管线无法从用户输入中确认当前在留资格类型，且在留类型对路由和风险判断有直接影响。
  常见触发：用户只说"我是签证"而非具体类型，或用户描述的工作/生活情况与签证类型存在多种可能性。

user_copy: "你现在持有哪种在留资格？"

follow_up_options:
  - "技术・人文知识・国际业务"
  - "経営・管理"
  - "技能"
  - "技能実習 / 育成就労"
  - "特定技能（1号・2号）"
  - "高度専門職"
  - "留学"
  - "家族滞在"
  - "日本人の配偶者等 / 永住者の配偶者等"
  - "定住者"
  - "永住者"
  - "その他 / 不确定"

what_info_needed: |
  在留资格类型（在留卡上的"在留資格"栏）。
  如果用户不确定，引导其查看在留卡正面中间区域。

internal_note: |
  不同在留资格类型的风险维度和可处理手续差异极大。
  技人国 vs 经管 vs 家族滞在在更新条件、工作限制、转换路径上完全不同。
  此问题是在留风险判断的最基础前提，缺失时不应假设任何默认值。
  如用户选"不确定"，应给出"可以拍在留卡照片确认"的提示。
```

### CL-02 — 在留期限/到期日不明

```yaml
clarification_code: CL-02
trigger_condition: |
  用户提问涉及更新、续签、材料准备，但未提供在留期限到期日，
  且期限距离影响风险级别判断（紧迫性差异）。

user_copy: "你的在留期限大概什么时候到期？"

follow_up_options:
  - "1 个月以内"
  - "1–3 个月"
  - "3–6 个月"
  - "6 个月–1 年"
  - "1 年以上"
  - "不确定 / 需要查在留卡"

what_info_needed: |
  在留卡上的"在留期間満了日"（到期日）。
  如用户选择"不确定"，引导其查看在留卡右下角区域或查询入管 online 系统。

internal_note: |
  期限剩余时间直接决定风险级别（HIGH/MEDIUM/LOW）和行动建议的紧迫程度。
  1 个月以内属于高风险，需要立即给出 handoff 提示。
  6 个月以上属于低风险，给出时间节点提示即可。
  此问题在涉及更新、材料准备、转换在留资格等场景均为必要信息。
  不得在不知道期限的情况下假设"时间充裕"。
```

### CL-03 — 申请状态不明（已申请/未申请）

```yaml
clarification_code: CL-03
trigger_condition: |
  用户提到更新或申请，但未说明是否已经提交申请，
  且已申请状态 vs 未申请状态的风险判断和行动建议完全不同。

user_copy: "你现在有没有已经提交更新申请？"

follow_up_options:
  - "还没有提交"
  - "已经提交了，在等结果"
  - "提交了但收到了补充材料通知"
  - "不确定"

what_info_needed: |
  是否已向入管递交了申请，以及申请处于哪个节点。
  如已提交：是否已拿到在留申請等待状態的证明（特例期间内的法律地位）。
  如已收到补充材料通知：通知上的期限。

internal_note: |
  已在特例期间（申请中）和未申请的法律处境完全不同。
  特例期间内用户仍可合法在留和工作，即使在留卡显示已到期。
  未申请且到期 = 超期居留，属于 HIGH 风险，handoff_trigger 必须命中。
  收到补充材料通知 = 需要确认通知期限，属于 HIGH 风险，时间敏感。
  这三种状态不能混用行动建议。
```

### CL-04 — 就労状況不明（就労中/離職/内定）

```yaml
clarification_code: CL-04
trigger_condition: |
  用户提问涉及工作相关的在留状况，但未说明当前是否在职、
  且就劳状态对在留资格有效性判断（尤其是技人国、特定技能、经管）有直接影响。

user_copy: "你现在的工作状态是？"

follow_up_options:
  - "在职（当前有雇用合同）"
  - "已离职，还没有找到下一份工作"
  - "已离职，有内定（还没正式入职）"
  - "自己经营（公司法人代表）"
  - "其他 / 不确定"

what_info_needed: |
  当前雇用/就労状态，以及如果已离职：离职时间和是否有内定/再就职计划。

internal_note: |
  技人国等就労系资格和实际雇用合同直接绑定。
  离职后通常有"正当な理由"期间（目前约3个月），但不是所有情况都适用。
  离职超过一定时间未续约/再就职可能影响下次更新许可。
  内定 vs 已入职在材料要求和风险判断上也有差异。
  不得在不知道就労状态的情况下给出技人国更新的一般建议。
  如用户是"自己经营"，应路由到经管签证相关判断，而非技人国。
```

### CL-05 — 配偶者関係変化の時期不明

```yaml
clarification_code: CL-05
trigger_condition: |
  用户提到与配偶相关的在留变化（离婚、配偶去世、配偶工作状态变化、婚姻关系变化），
  但未说明变化发生的时间，且时间点影响在留处理的紧迫性和手续要求。

user_copy: "这个变化大概是什么时候发生的？"

follow_up_options:
  - "最近 2 周以内"
  - "1 个月以内"
  - "1–3 个月前"
  - "3–6 个月前"
  - "6 个月以上"
  - "不确定具体时间"

what_info_needed: |
  配偶关系变化的时间点（离婚日期、配偶死亡日期等），
  以及是否已向入管提交了相关的届出或在留资格变更申请。

internal_note: |
  以日本人の配偶者等、永住者の配偶者等、家族滞在为例：
  配偶关系变化后，持有者在留资格的法律基础发生变化。
  入管要求在变化后14日内提交相关届出，超期未提交有处罚风险。
  超过6个月未处理在某些情况下可能影响后续审查记录。
  时间距离决定：是否已超过届出期限、是否已超过变更申请时间窗口、是否需要 handoff。
  此场景 handoff_trigger 通常应命中（配偶关系变化 + 在留资格 = DOMAIN 专业判断范围）。
```

### CL-06 — 会社清算/経営状況不明

```yaml
clarification_code: CL-06
trigger_condition: |
  用户持有経営・管理在留资格，且提问涉及公司变化（解散、清算、停止经营、
  转让经营权、公司名称变更），但未说明具体的处理状态。

user_copy: "公司目前处于哪个状态？"

follow_up_options:
  - "还在正常经营"
  - "已经停止经营（公司还没有解散登记）"
  - "正在进行清算手续"
  - "已经完成清算，公司已解散"
  - "准备转让给他人经营"
  - "其他 / 不确定"

what_info_needed: |
  公司的法人登记状态和实际经营状态，以及用户是否已经针对公司变化向入管提交了届出或变更申请。

internal_note: |
  経営・管理在留资格与经营实体的存续直接绑定。
  公司清算或长期停业可能导致在留资格失去存续基础。
  不同状态（停业 vs 清算中 vs 已解散）的手续和风险完全不同：
  - 清算中：仍有可能维持或转换资格，时间敏感
  - 已解散：可能需要立即处理在留资格出路（转换或离境）
  - 准备转让：经营管理资格通常不随法人转让，需要专门确认
  所有经管签证 + 公司变化的组合，handoff_trigger 通常必须命中。
  不得在不知道公司清算状态的情况下给出通用的经管更新建议。
```

### CL-07 — 离境目的不明（短期/長期/永久帰国）

```yaml
clarification_code: CL-07
trigger_condition: |
  用户提问涉及离开日本的计划，但未说明离境性质，
  且不同的离境目的对在留卡、居住地届出、公司手续、永住资格等影响完全不同。

user_copy: "这次离开日本，计划是？"

follow_up_options:
  - "短期出境（一两周，会回来）"
  - "中期离开（几个月，但会回来）"
  - "暂时不确定，可能会回，也可能不回"
  - "计划长期或永久回国，不打算再续签"

what_info_needed: |
  离境的预期时长和是否打算继续维持在日在留资格。
  如果是永久回国：是否需要处理公司清算、在留卡返纳、海外转出届、纳税管理人等事项。

internal_note: |
  短期出境和永久回国的处理事项完全不同：
  短期：注意不要超过在留卡有效期、再入国许可/特别再入国许可问题
  中期：如超过 1 年可能失去永住资格的居住实质；如超出再入国许可期限，在留资格消灭
  永久回国：需要处理的事项清单极长（公司清算、纳税管理人、海外転出届、在留卡返纳、各保险注销等）
  "不确定"状态：重点帮用户识别如果回国会影响哪些现有在留状态和资产（如进行中的永住申请）
  此场景 DOMAIN-CC 判断永久回国相关手续为高复杂度，handoff_trigger 通常应命中。
  用户问题如涉及"放弃经管签回国"，必须触发完整的手续清单提示，不得只回答其中一件事。
```

## 追加澄清规则

### 当需要追加第二个 clarification 时

如果第一个 clarification 回答后，管线仍然需要更多信息，允许追加一个 clarification，但：

1. 每次只追加一个
2. 追加前必须先给出"基于当前已知信息的初步说明"，再提出追加问题
3. 不允许连续追加超过 2 次后仍不给任何实质内容
4. 如追加 2 次仍无法路由，进入 `human_review_required` 状态，建议用户找行政书士

**示范：**
> 基于你目前提供的信息：在留期限还有 2 个月，还没有提交更新申请。
>
> 在确认建议步骤之前，还需要了解一件事：你的在留资格类型是？

---

### 永远不问的 clarification

以下信息不允许通过 clarification 请求收集，原因是风险过高或超出 TEBIQ 当前处理能力：

| 禁止收集的信息 | 原因 |
|----------------|------|
| 护照号码 | 个人证件号，TEBIQ 不需要 |
| 在留卡号码 | 个人证件号，TEBIQ 不需要 |
| 具体收入金额 | 隐私敏感，超出当前处理能力 |
| 家庭成员详细个人信息 | 超出当前处理能力 |
| 是否有违法记录 | TEBIQ 不具备处理此类信息的能力 |
| 雇主公司详细名称和地址 | 超出必要信息范围 |
