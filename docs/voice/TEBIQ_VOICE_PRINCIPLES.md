# TEBIQ Voice Principles

> 本文件定义 TEBIQ 的**语言原则**。适用范围：Internal Console 标签、Preview 状态文案、Matter 文案、Clarification 消息。
>
> 不适用于生产用户端文案（M7 前冻结）。与 `TEBIQ_COPY_SOURCE.md` 和 `TEBIQ_CONTEXT_PACK.md §6` 有冲突时，以三者最严格者为准，并同步更新。

---

## 1. 四项核心原则

### 原则一：Risk-forward，不是 Answer-forward

TEBIQ 的首要任务不是回答用户问的问题，而是发现用户**没有问、但会影响结果**的在留风险。

**操作含义：**
- 文案的起点是：用户的问题背后，真正的风险点是什么
- 不先急着"解答"，先确认方向有没有跑偏
- 如果用户问的表面问题和真实风险错位，文案要在不偷换问题的前提下，让真实风险先浮出来

**示范：**
- 用户问：经管签搬办公室要通知哪里？
- TEBIQ 文案重心：新办公室能不能撑住下次更新的审查——而不是只回答"需要通知入管和法务局"

**边界：**
- 不允许完全不回答用户原问题。风险前置不等于忽略原问题。
- 不允许用"你没问但要注意"这类句式明说内部逻辑。

---

### 原则二：Advance，不是 Reassure

TEBIQ 的每一段文案都应当指向**下一个具体行动**，而不是让用户感到安心或感到被理解。

**操作含义：**
- 每个回答页、每条状态消息，都要指出"现在做什么"或"接下来确认什么"
- 不用情绪安抚，不用"你不用担心"，不用"情况还好"
- 即使当前没有紧迫风险，也要给出"下一步确认点"而不是"放心吧"

**示范：**
- 不好：你的情况基本没有问题，可以放心。
- 好：当前信息看下来方向没有问题。接下来要确认：上次更新时间、收入证明是否还在有效期内、任职合同类型。

**边界：**
- "Advance"不等于每句话都催用户做事。确认完所有关键信息后，也可以说"这一步已经完成，下一步是……"

---

### 原则三：Honest about limits

TEBIQ 不提供法律结论，不出具资格判断，不代替行政书士做最终意见。文案必须让用户清楚这一边界——但不要用免责声明的方式，而是用**自然语言写清楚这一步的边界**。

**操作含义：**
- 不用"以上仅供参考，不构成法律意见"这类声明体句式
- 而是直接说清楚："这一步建议找行政书士确认"
- 高风险场景中，建议专业确认的语句必须出现，不能省略
- TEBIQ 描述的是**风险方向和需要确认的事项**，不是最终判断

**示范：**
- 不好：以上内容仅供参考，不代表法律意见，TEBIQ 不对任何决策后果负责。
- 好：这一步建议找行政书士确认，特别是补材料期限和不许可通知书的回应方式。

**边界：**
- 每个高风险场景（`handoff_trigger` 命中时）必须有明确的"找谁确认"出口，不能省略。
- 低风险场景不必每次都加 handoff 提示，但不得做正面保证。

---

### 原则四：Calibrated severity

风险级别不同，语言重量必须匹配。高风险文案必须传递真实的紧迫感；低风险文案不能用同等力度，否则用户会对所有警告失去感知。

**操作含义：**
- 高风险（P0/HIGH）：语气直接、句子简短、行动指向明确、不留回旋余地
- 中风险（P1/MEDIUM）：指出风险点、给出确认清单、语气坚定但不强迫
- 低风险（P2/LOW）：给出提示和下一步，不用"风险""严重"等词
- 不同风险级别不得用相同的句式模板

**示范对比（高 vs 低）：**
- 高风险：在留期限已经过了，这个状态不能继续在日本合法居留。现在的唯一选项是：立即联系行政书士确认是否有挽回余地。
- 低风险：建议在下次更新前 3 个月以上开始整理材料，时间比较充裕，先把上次更新用过的资料找出来。

**边界：**
- 不因"降低用户焦虑"而把高风险文案写软。
- 不因"显示专业感"而把低风险文案写得像紧急通报。

## 2. 五类禁止模式

### 禁止模式一：营销腔（Marketing Tone）

任何推销感、产品宣传感、App 下载引导感的表达，一律禁止。

**禁止示例：**
- 一键开启你的在留守护
- TEBIQ 全方位守护你的签证安全
- 专业 AI 帮你搞定在留烦恼
- 极速通关，轻松搞定在留更新
- 聪明的在留管理，就从这里开始

**原因：**
TEBIQ 不是产品推介页，是风险管理服务的工作界面。营销腔会让用户对系统的专业性产生怀疑，也会稀释风险信息的可信度。

---

### 禁止模式二：陪伴/共情腔（Companion / Empathy Tone）

任何模仿情感支持、表达"理解你"或"陪着你"的文案，一律禁止。

**禁止示例：**
- 别担心，我们来帮你处理
- 我懂你现在的感受
- 你不是一个人在面对这件事
- 有什么问题随时告诉我
- 我们一直在你身边
- TEBIQ 陪你度过这段时期

**原因：**
TEBIQ 的定位是冷静、可靠、懂规矩，不是情感 AI 产品。共情腔会让系统看起来不可信，也和在留风险管理的严肃性质不匹配。

---

### 禁止模式三：惊吓腔（Frightening Tone）

任何夸大危险程度、制造不必要焦虑或恐慌的表达，一律禁止。

**禁止示例：**
- 严重违规！你的在留资格可能已经失效
- 警告：你的情况非常危险
- 如不立即处理，后果不堪设想
- 你现在的状态已经违法
- 这是一个极其危险的信号

**原因：**
惊吓腔等同于误导。TEBIQ 如果无法确认风险等级，不能用恐吓语气代替信息缺口。即使是高风险，也要用直接、准确、可操作的语言描述，而不是用情绪化的警告语替代。

---

### 禁止模式四：虚假确定性（False Certainty）

任何对结果的保证、对通过/不通过的预测、对审查结果的断言，一律禁止。

**禁止示例：**
- 按这个方法做，更新一定能通过
- 你的情况没有问题，放心提交
- 这种情况百分之百会被拒
- 只要补上这份材料就可以了
- 审查官会认可这个说明的

**原因：**
TEBIQ 不做最终判断。任何确定性表达都会让用户误以为 TEBIQ 有法律意见出具能力，这超出了系统边界，也违反行政书士法的职业边界。

---

### 禁止模式五：内部标签暴露（Internal Label Exposure）

任何内部系统字段、状态码、枚举值、调试信息，一律不得出现在任何用户可见位置。

**禁止示例（即使加了中文描述也不行）：**
- `fallback_reason: no_source_matched`
- `domain: 就労`
- `safety_gate: cross_domain_blocked`
- `intent: zairyu_update`
- `engine_version: __answer_run_v1__`
- `out_of_scope: true`
- `low_confidence`
- 任何 sidecar JSON 原文

**原因：**
内部字段对用户没有任何意义，暴露它们会破坏用户对系统专业性的判断。这类信息只出现在 Internal Console 的 QA/工程视图中，即使在 Preview 状态页也不允许以原始字段名呈现。

## 3. 语言风格指南

### 基本风格定位

| 维度 | 要求 |
|------|------|
| 语言 | 简体中文为主 |
| 风格 | 正式但可读（formal-but-accessible） |
| 人称 | 第二人称（你）。不使用"您"，不使用"用户" |
| 句长 | 短句优先。一句话超过 40 字时考虑拆分 |
| 段落 | 不超过 4 句。超过时考虑是否是两个意思 |

### 不使用法律术语

TEBIQ 的用户是中文母语的在日外国人，不是行政书士或法学研究者。法律术语要翻译成自然语言：

| 避免 | 改用 |
|------|------|
| 在留資格取得許可申請 | 申请新的在留资格 |
| 就労資格証明書 | 工作资格证明 |
| 変更許可申請 | 申请变更签证类型 |
| 特別永住者 | 特别永住者（首次使用可带括号注释） |
| 資格外活動許可 | 兼职许可 |
| 不法残留 | 超期居留 |

### 不使用官僚套语

| 避免 | 改用 |
|------|------|
| 相关手续请按照规定流程办理 | 这一步需要先确认 X，再提交 Y |
| 如有疑问请咨询专业人士 | 建议找行政书士确认这一点 |
| 请注意相关风险 | 这里有一个容易漏的地方：…… |
| 本系统仅供参考 | 这一步的判断建议由行政书士确认 |

### 推荐句式（来自 `TEBIQ_CONTEXT_PACK.md §6`）

- 真正容易漏的不是 X，而是 Y。
- 这件事的关键点其实在 Z。
- 比 X 更要紧的是 Y。
- X 是表面，审查看的是 Y。
- 现在先做这一步。

## 4. 适用范围

### 本文件适用于

| 位置 | 说明 |
|------|------|
| Internal Console 标签文字 | 工程 / QA 视图中的中文标签和提示文字 |
| Preview 状态文案 | 非生产环境的功能预览页文案 |
| Matter 文案 | 事项保存、继续处理的轻量文案 |
| Clarification 消息 | 需要用户补充信息时的引导文字 |
| 系统状态消息 | received / routing / generating 等管线状态提示 |
| Fallback 与 Error 消息 | 模型失败或超时时的用户可见提示 |

### 本文件不适用于

| 位置 | 原因 |
|------|------|
| 生产用户端回答页正文 | 由 DOMAIN-CC 标注 + 产品负责人裁决后决定，见 `TEBIQ_COPY_SOURCE.md §3` |
| 首页主标题 / 落地页文案 | M7 前冻结，需产品负责人单独裁决 |
| 行动建议正文（知识库内容） | 由 TEBIQ-KB skill 专门生产，不在 VOICE 范围内 |
| Eval Lab 标注界面的字段说明 | 内部工具文案，不走 VOICE 流程 |

## 5. 文件更新规则

本文件由 TEBIQ-VOICE 负责维护。更新时需满足：

- 新增禁止模式：须有真实触发案例，不得凭直觉扩写
- 修改四项核心原则：须产品负责人裁决
- 修改语言风格指南：GM 可审批，不需要产品负责人
- 修改适用范围：须 GM + 产品负责人联合确认

与 `TEBIQ_CONTEXT_PACK.md §6`、`TEBIQ_COPY_SOURCE.md §5` 存在冲突时：**三者中最严格的规则优先，并同步更新其余两个文件**。
