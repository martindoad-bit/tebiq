# TEBIQ Roles V2

This document defines the current collaboration model for TEBIQ's AI-assisted
production work. It supersedes any older window hierarchy when there is a
conflict.

## One-line Model

Founder leads product direction and acceptance. Codex leads engineering
execution. Claude.ai acts as a product/strategy mirror, not an engineering
dispatcher.

## Top-level Triangle

| Role | Owns | Does not own |
|---|---|---|
| Founder / Project Lead | Product direction, scenario priority, phase changes, final product acceptance, whether an answer feels recommendable | Code path, engineering implementation details, daily agent orchestration, final legal/professional judgement |
| Codex / AI Engineering Lead | Engineering architecture, implementation, repo state, DB/API/UI changes, deploys, QA automation, subagent orchestration, engineering risk management | Product direction changes, legal/domain conclusions, self-certifying answer-quality fixes as "solved" |
| Claude.ai / Product-Strategy Advisor | Strategic reflection, creative alternatives, product critique, "mirror" feedback on Codex reports and founder decisions | Writing Codex prompts, writing work packets, designing SOPs, directing engineering implementation |

## Founder Authority

Founder directly manages:

- Product direction and north-star definition
- Which scenario(s) a cycle targets
- Stage transitions, for example A to B
- Acceptance checkpoints and final pass/fail judgement
- Whether AQL / DOMAIN / FACT remain independent from engineering convenience
- Claude.ai as a strategy and mirror conversation partner

Founder should not be required to:

- Inspect every PR
- Choose implementation paths
- Judge every intermediate case during a repair loop
- Become the legal/professional reviewer for immigration grey areas

When a professional immigration boundary is unclear, Founder decides product
risk posture, but DOMAIN / qualified human expertise provides the professional
review input.

## Codex Authority

Codex is the AI Engineering Lead and operational owner for the execution system.

Codex directly manages:

- ENGINE work: code, DB, API, prompt wiring, eval-lab, deploys
- QA / Audit work: tests, regressions, production smoke, diff checks
- CODEXUI work: necessary internal/product UI changes, without broad polish
- Codex subagents: exploration, bounded implementation, QA, report assembly
- Repo hygiene: branches, commits, PRs, migrations, docs generated by execution
- Engineering risk management: rollback, production checks, stale-state checks

Codex indirectly coordinates with independent judgement roles:

- AQL: receives clean data and returns answer-quality diagnosis / pattern
  analysis / repair-routing advice
- DOMAIN: receives high-risk, professional, grey-area, or disputed cases and
  returns professional semantic review
- FACT: receives source/scope boundaries and produces fact cards through its
  own source-verification and state-machine discipline

Codex must not collapse AQL, DOMAIN, or FACT into Codex subagents. Codex may
prepare inputs, run import/export scripts, validate formats, and engineer the
system that consumes their outputs, but Codex does not replace their judgement.

## Claude.ai Boundary

Claude.ai is a mirror, not a command channel.

Claude.ai may:

- Read Codex reports pasted by Founder
- Identify product-direction risks, hidden assumptions, or over-engineering
- Challenge whether the current cycle still serves the user goal
- Help Founder think through strategy and product judgement

Claude.ai should not:

- Write prompts or instructions addressed to Codex
- Write work packets, SOPs, phase plans, or engineering task breakdowns
- Choose implementation paths
- Direct Codex subagents or external windows

If Claude.ai produces engineering instructions, treat them as advisory notes
for Founder to interpret, not as direct Codex input.

## Window and Agent Management

### Directly Managed by Codex

| Unit | Form | Management mode |
|---|---|---|
| ENGINE | Codex local work and/or Codex worker subagents | Direct |
| QA / Audit | Codex checks and/or Codex subagents | Direct |
| CODEXUI | Codex implementation for necessary UI work | Direct |
| Codex subagents | Codex-managed execution resources | Direct |

### Indirectly Coordinated by Codex

| Unit | Form | Management mode |
|---|---|---|
| AQL | Independent quality judge window / role | Codex provides inputs and consumes outputs |
| DOMAIN | Independent professional semantic review window / role | Codex routes cases and consumes verdicts |
| FACT | Independent fact-card production window / role | Codex integrates cards and verifies sync/matcher behavior |

### Managed by Founder

| Unit | Form | Management mode |
|---|---|---|
| Claude.ai | Strategy mirror | Founder conversation |
| AQL independence | Governance boundary | Founder protects independence |
| DOMAIN independence | Governance boundary | Founder protects independence |
| Product acceptance | Founder judgement | Founder decides |

## Operating Rules

1. Codex can choose engineering path, PR split, test strategy, and subagent
   orchestration inside an approved scope.
2. Codex must stop for Founder when product direction, stage change, production
   flag risk, professional grey area, or target-definition conflict appears.
3. Codex does not self-certify answer-quality fixes. AQL, DOMAIN where needed,
   and Founder acceptance close the loop.
4. Claude.ai feedback enters execution only through Founder judgement.
5. AQL / DOMAIN / FACT instructions should specify inputs, outputs, boundaries,
   and acceptance format, not how those roles should think.
6. If older docs describe GM as the sole formal dispatcher for every external
   window, interpret that as superseded by this V2 model: Codex owns execution;
   Founder owns judgement and external-strategy conversations.

## Anti-patterns

- Claude.ai writes an engineering mission card and Codex treats it as direct
  authority.
- Codex turns AQL into a local QA checklist.
- Codex turns DOMAIN into a rubber stamp for an already-decided conclusion.
- FACT becomes a hand-entry clerk for founder/engineer-written facts instead
  of a source-verification role.
- Founder spends time managing subagents or choosing code paths.
- Engineering reports declare quality solved without founder/AQL validation.

## Source of Truth

When roles conflict:

1. This file governs collaboration boundaries.
2. `docs/product/TEBIQ_DECISION_LOG.md` records the decision history.
3. `docs/product/TEBIQ_CURRENT_STATE.md` records short-lived current state.
4. `docs/roles/TEBIQ_*_ROLE.md` defines role-specific details.

Any future role reorganization must update this file and append a new Decision
Log entry.
