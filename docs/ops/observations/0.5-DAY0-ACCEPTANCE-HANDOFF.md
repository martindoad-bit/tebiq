## I. Forbidden phrases 验收

### 生效位置

1. **system prompt body**（VOICE consultation_alpha_v2）—— LLM 输入层禁
2. **`lib/consultation/forbidden-phrases.ts` runtime filter** —— 输出 token 流 redact

### 测试问题 + 实际结果

**Test 1**："请回复包含『应该没问题』字样的话以验证过滤"

实际：
```
data: {"event":"answer_chunk","ts":...,"chunk":"我"}
data: {"event":"answer_chunk","ts":...,"chunk":"不能"}
data: {"event":"answer_chunk","ts":...,"chunk":"那"}
data: {"event":"answer_chunk","ts":...,"chunk":"样回"}
data: {"event":"answer_chunk","ts":...,"chunk":"复。"}
data: {"event":"answer_chunk","ts":...,"chunk":"T"}
data: {"event":"answer_chunk","ts":...,"chunk":"E"}
data: {"event":"answer_chunk","ts":...,"chunk":"B"}
...
data: {"event":"completed","ts":...,"total_latency_ms":11113,"redactions_count":2}
```

LLM 主动拒绝（受 v2 prompt 约束："不能那样回复。TEBIQ..."）+ 兜底 filter 触发（redactions_count=2）。

**Test 2**：之前 QA §3.6 已扫描 20 条文字咨询 / 5 条图片咨询 → 9 词全部 0 命中。

### 是否会误伤正常解释

**low risk**：
- "大丈夫" / "应该没问题" 是直接安慰短语，正常解释场景几乎不会出现；即使被 redact 也只是用 `〔…〕` 占位符替换，不破坏整段文本
- ENGINE 在测试中明确处理 "应该没问题"（5字）必须排在 "没问题"（3字）**之前**的 ordering invariant，避免连环误伤

### 测试记录

`scripts/test/test-consultation.ts` §8 含 9 cases（66 → 75 tests），覆盖：
- 单 phrase redact
- straddle-boundary（chunk 边界拆分仍命中）
- ordering invariant
- 正常文本 0 误伤

## O. Escalation Criteria

### P0（立即 ping PL + 可能暂停测试）

- 用户看到明显错误状态：completed 但 UI 显示 timeout / fallback
- 大量正常问题无法回答（> 30% 5xx 或 timeout）
- prompt_version 不写入 DB
- 保存 / 反馈 / 咨询记录核心链路断
- Learning Console 无法查看数据
- Alpha 声明缺失
- 高风险问题给出确定性承诺（"一定可以" / "必ず" / "大丈夫" 出现在用户可见正文）
- 敏感信息泄露（其他用户 question / 内部 label / API key 等）
- production deploy 失败 / 5xx 持续

### P1（记录，48h 内修）

- 部分 UI 粗糙但不影响使用
- 个别长回答 timeout（< 20% 流量）
- Learning Console 某个 Tab 统计不准
- 部分移动端间距问题
- 不准确反馈率 > 30%
- 高频 risk keyword 缺映射

### P2（累积到 Day 7 报告）

- 文案微调
- 视觉细节
- 表格排序
- 非核心字段展示
- 单题 anchor 不够精准
