# Work Packet — 慢响应 / 失败 / 重试 状态体验 (CODEXUI + ENGINE 协作)

**对象**: CODEXUI (UI 主导) + ENGINE (SSE event 协作)
**任务标题**: 用户看到 "系统在工作 vs 系统死了" — 状态分层 + 失败/重试路径
**Sprint**: TEBIQ 0.6 / Wave 3
**Issued by**: GM (per PL 2026-05-07 §5 directive)

---

## 背景

PL §5 真实观察：

> 微信内打开最终可用，但有时等待 40-50 秒才出答案。这在内部测试阶段不是 P0，但会影响用户判断系统是否卡死。

> 慢可以，但不能像死了一样慢。用户要知道系统仍在处理、可以继续等，或已失败可重试。

Pro thinking 模型 first_token 4-16s 是常态；偶尔可达 30-45s。当前 routing_status 提供 0ms + 3000ms 两层文案（Pack 1 已落地），但 15s+ 后用户仍无任何状态变化 → 焦虑。

## 当前问题

1. **15-20s 静默期**: 收到 `routing_status` specific 后到 first_token 之间长达 10-20s 无 UI 反馈
2. **30-45s 超时态**: 用户不知道是继续等还是已死
3. **真失败路径不清**: SSE `failed` / `timeout` 事件 UI 处理可能与 completed 混淆
4. **重试入口不明显**: 失败后用户不确定要刷新还是按某按钮
5. **微信内置专项 smoke 缺失**: 需要专门验证微信环境下状态显示是否正确

## 目标

```
慢可以，但不能像死了一样慢。
用户要知道系统仍在处理、可以继续等，或已失败可重试。
```

具体阶层目标（阈值由 ENGINE/CODEXUI 实测后决定，PL 不规定具体秒数）：

- 0-1s: received / 已收到 状态显示
- 3-5s: routing_status specific 显示（已落地 PR #79）
- 15-20s 仍无正文: 显示更明确的"仍在生成"状态
- 30-45s 仍无正文: 给用户明确选择 — 继续等待 / 稍后回来 / 重试
- 真失败: 进入 failed / retry 状态，不要无限等待
- 微信内置浏览器专项 smoke

## 工作方式

### CODEXUI（UI 主导）

PR description 顶端 5 问 UX 判断：

```
1. 用户在 15-20s 静默期会想什么？文案应该传递什么？
2. 30-45s 时给"继续等待 / 稍后回来 / 重试"3 选 1，还是默认继续 + 一个 escape hatch？
3. 真失败 vs 超时 vs 用户主动取消 — UI 上区分还是合并？
4. 重试是 in-place 还是跳新 stream？
5. 微信内置浏览器有什么不同的状态表达需要（如不能显示动画？）
```

实现 UX：
- 0-1s + 3-5s 状态保留（PR #87 已落地）
- 15-20s 后新增 "仍在生成" 状态（CODEXUI 自决文案）
- 30-45s 后给"继续等待 / 稍后回来 / 重试" 入口（CODEXUI 自决形态）
- failed / timeout / partial 状态切换清晰（per UX refinement 教训）
- 微信内置专项 smoke

### ENGINE（SSE event 协作）

如果 CODEXUI 判断需要新 SSE event 来标记 "long_wait" 状态（如 15s 后发出 `long_wait_status`），ENGINE 配合落地：
- 在 stream route 加 timer
- 不影响 first_token latency
- 默认 enable，无需 feature flag（属于 UI 提示，无 user-facing 数据风险）

如果 CODEXUI 判断仅靠现有 events（received / routing_status / first_token）+ 客户端 timer 即可，ENGINE 不动后端。**这是 CODEXUI 的判断**。

ENGINE 也确认：现有 `failed` / `timeout` event payload 是否包含足够信息让 CODEXUI 区分（如 `error_code` / `recoverable: bool`）。如果不够，ENGINE 加。

## 约束

不做：

- ❌ 不改模型（已默认 deepseek-v4-pro thinking）
- ❌ 不改 latency baseline（不让等待状态本身拖慢回答）
- ❌ 不显示 fake progress（如假装 20% / 50% 进度条 — 不真实）
- ❌ 不显示 reasoning_content（thinking 模型内部不暴露）
- ❌ 不在第一次打开就强推"添加到桌面"（per save/share refinement）
- ❌ 不让 completed / partial / failed / timeout 状态混淆（per UX refinement 教训）
- ❌ 不绕过 GM 联系其他窗口

可做：

- ✅ CODEXUI 加新 UI states / 文案 / 视觉
- ✅ ENGINE 加新 SSE event（如 long_wait_status）
- ✅ 客户端 setTimeout 触发"仍在生成"显示（如 15s timer）
- ✅ 重试按钮触发新 POST /api/consultation/stream（with same question）
- ✅ "稍后回来" 引导用户保存（与 save/share 路径联动）

## 验收 (12 项)

1. 0-1s 有 received / 已收到状态
2. 3-5s 有具体 routing_status (已落地)
3. 15-20s 仍无正文时显示更明确的"仍在生成"状态
4. 30-45s 仍无正文时给用户明确选择 (继续等 / 稍后回来 / 重试)
5. 真失败进入 failed / retry，不无限等待
6. 微信内置浏览器专项 smoke 通过（截图）
7. 状态不与 completed / fallback 混乱
8. completed / partial / failed / timeout 各自 UI 表达清晰
9. 重试入口语义清楚（不混淆"重试" vs "新问题"）
10. 不影响 first_token p50 latency
11. UX 判断段在 PR description 顶端
12. 6 breakpoint 截图（含微信内置专项）

## 完成回报

CODEXUI 19 字段 + UX 判断段。
