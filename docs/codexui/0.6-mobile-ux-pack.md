## 3. Workstream E — Mobile UI / 阅读体验

### 产品目标

手机上读起来像"专业咨询产品"，不是一大块 AI 文本。

### 3.1 首屏字号 + 行高 + 段落间距

- 移动端正文字号当前偏小 → 提升到适合手机认真阅读的水平（具体由你判断）
- 行高更舒展
- 段落之间留更多呼吸空间
- 不要变成 e-book 风格，仍要紧凑

### 3.2 答案层级（不要僵硬模板化）

答案视觉上应自然分成可识别的"关键点 / 需要确认 / 下一步"层次。**但不要做成统一模板**，因为模型回答的结构会变化。

可考虑的实现路径（你选）：
- markdown 渲染优化（标题 / 列表 / 强调样式）
- 段落间分割线
- 关键句视觉强调（如左侧 border + 加粗）
- 末段 "下一步建议"自然识别

不要做：
- 强制每个回答都有"问题摘要 / 答案 / 下一步"三段式 header
- 把 AI 回答塞进固定 schema 卡片

### 3.3 首屏输入效率

- 减少非必要占位
- 输入框立即可用，不需要滚动
- 拍照入口和文字输入形成自然的"上传材料一起问"入口（不要埋在最下方）
- "记录" 按钮文案改成更明确的 "我的咨询记录" 或你判断的更好表达

### 3.4 风险提示与回答卡片的关系

- 风险提示当前是泛泛文案 → 命中关键词后会有更贴近问题的（Workstream G 后端会补 keyword bucket → 风险维度映射）
- UI 上克制，不要变成吓人弹窗
- 风险提示放在回答上方还是下方由你判断；只要不抢主任务

### 3.5 回答完成后的动作

回答底部清晰提供（Workstream F 部分）：

- 保存这次咨询
- 复制链接
- 分享给自己（系统 share sheet）
- 补充情况（Workstream D 接入后）

### 3.6 Alpha 声明

保留，但视觉上更克制，不要抢主任务。

### 3.7 不要做（Workstream E 边界）

- 不重做品牌
- 不改 logo
- 不换主色
- 不做花哨插画
- 不进入完整 App 风格重构
- 不影响当前回答速度

## 4. Workstream B — First-response 等待状态 (UI 部分)

### 背景

Pro thinking 默认下，first_token 可能 4-16s。用户不能觉得页面卡死。

### 后端约定（GM 后续 PR 落地）

后端会在 SSE 流中按以下时序发出事件（这些事件名以 `docs/engineering/0.6-fact-layer-design.md` 为准）：

| 事件 | 时机 | payload |
|---|---|---|
| `received` | 0 ms | `{consultation_id, ts}` |
| `risk_hint` | <100 ms | `{risk_keyword_hits: string[]}` |
| `routing_status` | 0-1s | `{bucket: string, status_label: string}` — keyword bucket 命中文案 |
| `routing_status` (升级) | 3-5s 仍无正文 | 第二层更具体的 status_label |
| `fact_cards_injected` | first_token 前 | `{items: [...]}`（详 design doc） |
| `first_token` | 模型首字 | `{first_token_latency_ms}` |
| `answer_chunk` | 持续 | `{chunk: string}` |
| `completed` / `partial` / `failed` / `timeout` | 末端 | … |

### 你的任务

#### 4.1 0-1s 状态层

收到 `received` 事件后立即显示：

> 已收到，正在整理这个问题涉及的在留方向。

文案具体由你决定，符合"不安慰、不陪伴、给方向"voice。

#### 4.2 3-5s 状态层

3-5s 内如果还没收到 `answer_chunk`，根据已收到的 `routing_status.status_label` 显示更具体的状态。

后端会按 keyword bucket 给如：

- 経営管理：「正在核对经营管理、在留资格变更和当前基准」
- 年金/税金：「正在核对缴纳记录、更新/永住影响和下一步处理」
- 技人国/工作内容：「正在核对在留资格活动范围和工作内容匹配」
- 配偶/离婚：「正在核对身份变化、届出和在留衔接风险」

文案具体值后端给 — 你只负责显示。

#### 4.3 正文出现后

`answer_chunk` 开始流入后，状态自然退场或弱化。不要让状态条和正文同时占视觉重心。

#### 4.4 超时态

如果超过合理时间（建议 30s）仍未 first_token，再显示：

> 仍在生成，可以继续等待。

#### 4.5 不要做

- 不要伪造模型结论（只显示状态，不显示推断答案）
- 不要说"正在查询官方数据库"，除非 `fact_cards_injected` 真的命中并 payload 里 `decision: 'inject'`
- 不展示 reasoning_content
- 不显示 "AI 正在思考"之类无信息量文案
- completed / partial / failed / timeout 状态切换必须清晰，不能混乱

## 5. Workstream F (UI 部分) — 保存 / 分享 / 回访

### 产品原则

- **不**把"添加到主屏幕"作为主路径
- 主路径：保存咨询 → 复制链接 → 系统 share → 我的咨询记录
- 桌面/PWA 引导只是次级帮助入口

### 5.1 回答完成后底部动作

清晰列出（你决定具体 layout）：

- 保存这次咨询
- 复制链接
- 分享给自己（用 navigator.share，fallback 复制链接）
- 补充情况（Workstream D 接入后激活；现在可灰显或先不出）

### 5.2 文案方向

- "这次咨询可以保存，下次继续查看。"
- "可以把链接发给微信文件传输助手、LINE、邮件或备忘录。"

具体措辞由你决定，遵守 voice anchor。

### 5.3 微信内置浏览器

可正常使用就保留。**不要强制跳 Safari**。微信里 navigator.share 通常不可用，复制链接为主。

### 5.4 "记录" 按钮

改成更明确的 "我的咨询记录" 或 "查看已保存"。

### 5.5 根路径

GM 会单独检查 `tebiq.jp/` 根路由是否能落到咨询主入口。CODEXUI 不需要改路由，但首屏 UI 要确保根路径直接看到咨询入口。

### 5.6 添加到桌面

本轮**不作为主路径**。

只做一个轻帮助入口，例如"如何下次快速打开 TEBIQ？"，里面顺序：

1. 保存/复制本次咨询链接
2. 发给自己或收藏
3. Safari 添加到主屏幕
4. 微信中可收藏或转发给自己

不要在第一次打开时弹出"添加到桌面"。

## 7. 你必须避免的错误

- ❌ 一个 PR 同时改 E + B + F（review burden 过大）
- ❌ 在 production 路径放置任何阻塞的 await（speed 不能恶化）
- ❌ 假装命中 fact_card 但实际后端未发 `fact_cards_injected`
- ❌ 把 reasoning_content 渲染出来
- ❌ 在 microcopy 里写承诺性文案
- ❌ 修改 `lib/answer/`, `app/api/`, `drizzle/`, `lib/voice/` 之外的内容
- ❌ 删除 Alpha 声明
- ❌ 自动跳转外部浏览器
- ❌ 在加载阶段渲染 placeholder 答案文本
