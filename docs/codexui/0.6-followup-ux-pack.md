# Work Packet — CODEXUI 受控补充 / Follow-up UX

**对象**: CODEXUI
**任务标题**: 用户语义连续性 — "补充" vs "新问题" 区分 + 上下文继承体验
**Sprint**: TEBIQ 0.6 / Wave 3
**Issued by**: GM (per PL 2026-05-07 §3 directive)
**Pre-condition**: ENGINE Pack 2.3 落地（提供 `/api/consultation/follow-up` endpoint + summary 机制）。本 Pack 设计可与 ENGINE 并行，UI 实现等 endpoint。

---

## 背景

PL §3 观察：

> "继续补充情况"看起来像在原咨询下追加背景，但实际体验像回到新问题页面，用户会觉得上下文断掉。这会破坏咨询产品感。

PL 之前对 UI 派工的修正（DELEGATION_PRINCIPLES）适用于本任务：**不要给按钮清单。给问题、目标、约束、验收，让 CODEXUI 自决信息层级 / 视觉表达**。

## 当前问题

1. 用户读完答案后，"继续补充情况"按钮点击 → 视觉上像回到首页输入框 → 心理上感觉这是新问题
2. 上下文断裂感 → 用户不确定系统是否记得上一轮答案
3. 缺乏明确的 "补充" vs "新问题" 区分 → 用户混淆
4. 多轮补充后没有 graceful 退出（PL §3 验收 5：超过 3 轮引导保存 / 人工 / 新问题）

## 目标

```
用户读完回答后能围绕同一个咨询事项补充背景；
系统应继承上一轮上下文，而不是把它当成全新问题。
```

具体：

1. 用户能明显感知"这是补充当前咨询"，不是"重新提问"
2. 视觉上让用户看到上一轮答案/咨询主题仍在场（不是清屏跳到新页）
3. 不变成普通无限多轮 AI 聊天
4. 默认最多 3 轮补充
5. 4 轮触发引导：保存 / 人工确认 / 新问题
6. "新问题" 入口与 "补充" 入口语义清楚
7. 补充链路不明显拖慢当前回答

## 后端契约（从 Pack 2.3）

CODEXUI 消费的内容（ENGINE 已实现 / 在实现中）：

- `POST /api/consultation/follow-up` with `{ parent_consultation_id, user_addition }` → SSE 流（同主 stream 形态）
- SSE event `follow_up_limit_reached` (4 轮触发) — payload: `{ message: string }`
- `parent_consultation_id` 在 response 头部或 SSE first event 提供（CODEXUI 用此 chain）
- consultation row 含 `consultation_summary jsonb` + `follow_up_count` + `parent_consultation_id` — `/c/[id]` 详情页可读

## 工作方式（CODEXUI 自决，先做 UX 判断）

按 PL §7 problem-statement 模式。在 PR description 顶端写 5 问 UX 判断：

```
1. 用户读完答案后，"补充" 心理状态 vs "新问题" 心理状态如何区分？
2. 上下文继承感如何视觉表达（保留答案/主题/已知事实）？
3. 4 轮上限触发时如何 graceful 引导（保存 / 人工 / 新问题）？
4. 补充输入框如何区别于首页输入框（避免"回到新问题"误感）？
5. 移动端补充链路如何不打断阅读流？
```

然后才动 UI 代码。

## 约束

不做：

- ❌ 不变成无限聊天产品（4 轮硬阻塞）
- ❌ 不堆功能控制台（per UX refinement 教训）
- ❌ 不暴露内部词（consultation_summary / fact_card 等不出现在用户端）
- ❌ 不重做 brand / Logo / 主色
- ❌ 不影响主回答 latency

可做：

- ✅ 重新决定 "继续补充情况" 按钮位置 / 形态 / 文案
- ✅ 重新决定补充输入框视觉（建议明显不同于首页输入框）
- ✅ 加上一轮答案的 reference / pin / collapsed-summary
- ✅ 移除 / 合并 / 折叠 现有 action area 中误导项
- ✅ 改写所有 microcopy

## 验收 (10 项)

1. 用户能明显理解："这是补充当前咨询"，不是"重新提问"
2. 补充输入框视觉与首页输入框可区分
3. 用户能感知系统继承了上一轮上下文（如显示上一轮 user_goal / 主题 reference）
4. 不变成普通聊天 UI（无气泡 timeline / 无"AI 助手" persona）
5. 4 轮上限到达时显示 graceful 引导（保存 / 人工咨询 / 新问题）
6. "新问题" 入口与 "补充" 入口语义清楚
7. 补充提交后 stream 状态 (received / routing_status / fact_cards_injected / answer_chunk / completed) 与主 stream 一致显示
8. 不暴露内部词
9. 微信内置 / iPhone Safari / Chrome / 390 / 430 / desktop 全部测试
10. UX 判断段在 PR description 顶端写明（PL §3.4 工作方式要求）
