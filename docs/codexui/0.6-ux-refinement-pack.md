## 派工原则修正（PL 明示）

旧 pack §3.5 / §5 用了"组件清单式派工"（加这个按钮、加那个按钮）。这导致了功能堆叠、主次不清、内部词泄露。

本 pack **不**给按钮清单。给的是：
- 用户场景
- 当前体验问题
- 产品目标
- 约束
- 验收标准

**CODEXUI 自行判断**：信息层级 / 按钮数量 / 主次关系 / 是否折叠 / 文案 / 布局 / 移动端节奏。GM 不替 CODEXUI 做视觉和组件决策。

## 1. 用户场景

用户在 `tebiq.jp/ai-consultation` 提了一个在留咨询问题，回答流式输出完成。
此时用户处于以下心智状态：

```
读完了一段比较长的、有判断的咨询答案。
情绪可能：松一口气 / 想再问一句 / 想保存 / 想以后再回来 / 想分享。
但首先 — 想消化刚刚读到的。
```

用户**不想**：被一堆功能按钮打断 / 被 onboarding 教育 / 被功能控制台包围。

---

## 2. 当前体验问题（PL 直接观察）

### A. 回答完成后的行动区过重

回答下面塞了：保存、复制、分享、记录、详情、快速打开、继续补充、反馈 ……
**像后台功能面板，不像咨询产品**。

### B. 用户端泄露内部系统词

例：UI 里出现 "反馈会进入 Learning Console"。
"Learning Console" 是 GM/PL 内部运营概念，**绝对不能出现在用户端**。
用户只需要知道 "反馈会帮助改进回答"，或更克制的表达即可。

类似要清扫的内部词（任何在用户端出现都需移除/改写）：
- Learning Console
- fact card / fact layer / 事实卡
- routing_status / SSE / event
- ai_verified / human_reviewed
- Sprint / Workstream / Pack
- internal / dry-run

### C. 主次关系不清

"保存这次咨询" / "继续补充" / "我的咨询记录" / "复制链接" / "分享给自己" 全部近似同级。
**当前场景下用户最自然的主动作是什么？** —— CODEXUI 自行判断。

### D. 功能堆叠感

部分区域像在说"我们把功能都做出来了"，而不是"用户下一步自然该做什么"。
TEBIQ 是冷静、专业、克制的咨询产品，**不是功能控制台**。

### E. 移动端阅读仍需优化

字号 / 行高 / 段落长度 / 卡片密度 / 按钮重量 / 首屏节奏。
CODEXUI 自行判断，GM 不规定 px。

---

## 3. 产品目标

```
用户读完回答后，知道接下来最自然的 1 个动作；
如果需要，也能找到保存、补充、回访方式；
但不会被一堆功能按钮打断。
```

核心：

1. 回答优先 — 操作区不能压过回答
2. 一个主动作清晰
3. 次动作克制 — 可折叠 / 弱化 / 推迟 / 移除
4. 不暴露内部系统词
5. mobile first
6. 微信内置浏览器 / iPhone Safari / iPhone Chrome 都要自然

---

## 4. CODEXUI 工作方式（先判断，再实现）

请 CODEXUI **不要直接按旧 pack 清单实现**。先做一段简短 UX 判断，落到 PR description 顶端：

```
1. 当前行动区为什么怪？
2. 用户读完回答后最自然的动作是什么？
3. 哪些按钮应该保留 / 弱化 / 合并 / 移除？
4. 哪些内部文案必须删除？
5. 移动端如何让回答和行动区更舒服？
```

判断不用长，但必须在 commit 之前先有。然后才动 UI 代码。

PR description 同时包含此次的设计判断 + 实现方式 + 19 字段完成回报。

## 5. 约束

不做：

- 不改品牌
- 不改 logo
- 不重做整站视觉
- 不扩大成 App
- 不新增复杂账号系统
- 不改模型
- 不影响回答速度
- 不暴露 Learning Console / internal / fact layer / Sprint / Workstream 等内部词
- 不机械保留所有旧按钮
- 不绕过 GM 联系 PL / FACT / DOMAIN / QA / ENGINE

可以：

- 移除现有按钮（如果 CODEXUI 判断不该出现）
- 合并按钮（如果两个动作可以合并为一个）
- 折叠次动作（隐藏到二级菜单 / 推到回答下方较远位置 / 设为辅助文案）
- 改写所有 microcopy
- 重新决定按钮形态（icon / text / link / banner）
- 重新决定动作触发位置（页面底部 / 浮动 / 侧栏 / inline）
- 重新决定首屏密度

## 6. 验收标准

GM / QA / PL 用以下检查 CODEXUI 调整后的 PR：

| # | 检查 |
|---|---|
| 1 | 手机上第一眼不乱 |
| 2 | 回答仍然是页面中心 |
| 3 | 用户知道下一步该做什么（一个主动作清晰） |
| 4 | 操作区不像后台控制面板 |
| 5 | 用户端没有内部系统词（Learning Console / fact / SSE / routing 等） |
| 6 | 主动作明确，次动作不抢 |
| 7 | 保存/回访路径存在，但不笨重 |
| 8 | 继续补充入口自然，但不变成普通聊天产品 |
| 9 | 微信内置浏览器可用 |
| 10 | iPhone Safari / Chrome 可用 |

GM 在 PR review 时按此 10 项 checklist 评。

## 8. 关于 Workstream B 等待状态（routing_status SSE）

ENGINE Pack 1 已 merge，`routing_status` SSE event 在 main 上可消费。

CODEXUI 可以在本次 refinement PR 一并处理 B-UI 等待状态显示，**或拆独立 PR**——CODEXUI 自行判断。

如果一并：UX 判断段需说明 0-1s + 3-5s + 30s+ 三层等待文案如何融入新设计（不是堆按钮风格的"loading 提示条"）。

SSE event payload 形态（ENGINE 已落地）：

```json
{
  "event": "routing_status",
  "ts": <ms>,
  "level": "initial" | "specific",
  "buckets": ["keiei_kanri" | "nenkin_zeikin" | "gijinkoku" | "spouse_divorce" | ...],
  "status_label": "正在核对经营管理、在留资格变更和当前基准。"
}
```

`buckets` 是数组（多桶命中），`status_label` 是 ENGINE 给的 top-1 文案 — UI 直接显示即可。CODEXUI 可决定是显示文案 / 仅显示进度 / 微动效 / 完全 invisible 进度。
