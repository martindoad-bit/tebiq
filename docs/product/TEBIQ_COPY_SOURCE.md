# TEBIQ Copy Source

> 本文件是**用户端文案与回答呈现原则** + 文案来源流程，不是单页文案候选库。
>
> 长期产品宪法见 `TEBIQ_CONTEXT_PACK.md`；当前主线短期状态见 `TEBIQ_CURRENT_STATE.md`。

---

## 0. 用途

- 定义用户端文案的**呈现原则**（如何展示 DeepSeek / TEBIQ 输出）
- 定义文案的**来源流程**（从 Eval Lab 标注 → 落地）
- 定义 Matter Draft 的语言边界
- 定义 DOMAIN / VOICE 在文案链上的关系
- 收录禁用词与已确认的"黄金方向"参考片段

使用规则：

- Claude Code / VOICE / UI / QA 在做用户端文案 / 表达任务时必须读取本文件
- 本文件不替代 `TEBIQ_CONTEXT_PACK.md`，两者必须同时读取
- 与 `TEBIQ_QA_GATES.md` 禁用词清单不一致时，以更严格者为准

## 1. 当前不冻结用户端最终文案

> **当前阶段，首页 / 回答页 / Matter / 30 秒检查的最终文案均不冻结。**

### 原因

按 Eval-driven 策略原则（`TEBIQ_CONTEXT_PACK.md §2`），用户端文案的最终形态应当**反推自 Eval Lab 标注结果**：

- 哪些场景下 DeepSeek 自然回答已经够好（→ 用户端不要包装）
- 哪些场景需要 TEBIQ 加 `must_have`（→ 用户端要补一句）
- 哪些场景需要 `should_handoff`（→ 用户端要明确指向行政書士）
- 哪些表达模式真的让用户读懂（→ 升级为 A 层文案）

**没有这批数据之前，任何宣称"已冻结的最终文案"都是猜的。**

### 副作用

- 不要为 [TBD] 字段补"占位文案"然后忘掉
- 不要在没有 DOMAIN-CC 标注的情况下，让 VOICE 一次写完整 6 屏
- 不要把"看起来不错的临时表达"提级为 A 层
- 任何"上线必须用这句话"的主张，必须先在 Eval Lab 标注里有对应支撑

## 2. 回答页当前原则

> 这一节定义**回答页应该呈现什么、不该呈现什么**。是产品级原则，不是某次 PR 的临时选择。

### 必须

- **保留用户原问题**。用户问的是 A，不许偷换成 A'。
- **第一版优先保留 DeepSeek 自然回答的可读性**：完整的中文段落、自然分段、不强拆。
- **TEBIQ 的加值（true_focus / 下一步）放在 DeepSeek 答完之后**，不要打断主回答。
- **高风险问题必须显式建议专业确认**（行政書士 / 入管 / 区役所 / 年金事务所等具体出口）。
- **明确边界**："以上是参考，不是法律意见 / 不是个案审查结论"。

### 不允许

- **不要展示"我理解你的问题是……"** 这类容易方向反的系统改写，**除非已经过方向校验**（即同时有 Eval 标注 `direction_correct: yes` 的支撑或人工方向核对）。如果没有这个支撑，就**直接展示用户原问题**。
- **不强行展示复杂 true_focus 标签 UI**。在标注积累足够前，true_focus 应以**自然语句**融入正文，而不是单独的标签 / 卡片 / icon。
- **不把答案拆成过多卡片**。一个连贯回答被拆成 4 张卡 = 包坏。
- **不显示内部字段**（`fallback_reason` / `engine_version` / `domain` / `intent` / 任何 sidecar JSON）。
- **不显示底层模型来源**：不出现 "DeepSeek" / "AI" / "GPT" / "ChatGPT" / "Claude" 字样。
- **不用禁用表达**："你没问但要注意"、"重要风险提醒"、"别担心"、"我懂你"、"我可以帮你"、"立即预约专家"、"AI 智能分析"、"一键开启专业守护"、"严重违规"。
- **不做确定性承诺**："一定通过"、"一定不通过"、"绝对没问题"等任何对结果的保证。
- **不发明事实**：手续名 / 期限 / 适用条件如不来自 DOMAIN-CC 标注的事实卡或 DeepSeek 原文，禁止由文案窗口补写。

### 灰色区与裁决

- "DeepSeek 答得好但缺一句安全边界" → ENGINE 在 surface safety 层补，不改主回答正文
- "DeepSeek 答错方向" → 由 Eval Lab 标注 `direction_correct: no` 触发；当前不展示，反推 prompt
- "DeepSeek 答得对但 TEBIQ 包坏" → P0（见 `TEBIQ_QA_GATES.md`），回退到展示原回答
- "用户问的是高风险题（不许可、补材料过期、可疑配偶离婚）" → 必须有 handoff 提示

## 3. 用户端 copy 的来源流程

> 用户端文案不再从空白 VOICE prompt 直接生成。

### 正确流程

1. **Eval Lab 产生问题和答案样本**（DeepSeek 裸答 + TEBIQ 当前管线输出）
2. **DOMAIN-CC 在 Eval Lab 标注**：`severity` / `direction_correct` / `must_have` / `must_not_have` / `should_handoff` / `action`
3. **产品负责人裁决**：哪些 `must_have` 应当进入用户端默认呈现、哪些情况升级为 handoff
4. **ENGINE 落地 copy**：根据裁决修改回答页 / Matter / 30 秒检查的呈现逻辑
5. **QA 做 visible text 和语义回归**：见 `TEBIQ_QA_GATES.md` 四层 QA

**禁止：**

- 跳过第 2 步（直接让 VOICE 写）
- 跳过第 3 步（让 ENGINE 自己决定哪些标注要落到用户端）
- 跳过第 5 步（自测就 merge）

### 文案分层（保留旧分层语言，但只 A 层算"已冻结"）

| 层级 | 含义 | 升级路径 |
|------|------|----------|
| **A 层 — 已冻结** | 经历过 Eval Lab 标注 + 产品负责人裁决 + Semantic QA 通过 | 由产品负责人裁决升级 |
| **B 层 — 待验证** | 当前可参考但未冻结 | 收到对应 Eval 标注后，要么升 A，要么改写 |
| **C 层 — 禁用** | 见禁用词与禁用模式（§5） | 永不升级 |

**当前 A 层为空。** 任何标记 [A] 的字段必须能指出对应的 Eval Lab 标注 / 产品负责人裁决记录；找不到的视同 B 层。

## 4. VOICE Task Card 模板

> VOICE 不常驻。仅在产品负责人 / TEBIQ-GM 给出完整 Task Card 时被调用。**没有 Task Card 不写文案。**

```md
# TEBIQ Voice Task Card

## 页面位置
例如：回答页 / 首页主标题 / 事项卡 / 风险结果页

## 用户原话
用户实际怎么问

## 用户状态
在留资格、到期日、最近变更、家庭关联、公司情况等

## DOMAIN-CC 标注
must_have：……
must_not_have：……
should_handoff：yes/no
direction：correct/uncertain
（如果没有，VOICE 应停止并要求补充，不要猜）

## 已确认事实
只放已确认事实，不放模型猜测

## 页面目标
这段文案要让用户做什么

## 正向样例
给 1–3 个好例子，并说明为什么好

## 反向样例
给 1–3 个坏例子，并说明为什么坏

## 禁用词
列出本任务禁用表达（含通用禁用词 + 本任务额外禁用）

## 输出要求
字数、格式、按钮数量、是否需要内部说明
```

### VOICE 独特约束

- VOICE 不允许决定页面结构
- VOICE 不允许自己发明法律事实
- VOICE 不允许一次写完整 6 屏
- VOICE 每次只处理一个页面或一个模块
- VOICE 必须基于 Context Pack + Task Card + DOMAIN-CC 标注 工作
- 没有上下文卡或没有 DOMAIN-CC 标注时，VOICE 应停止并要求补充，**而不是猜**
- VOICE 输出的文案必须标注层级（B 层），不能自行升级为 A 层

## 5. 禁用词与禁用模式

### 禁用词（出现在任何用户可见位置即为 P0）

- 别担心
- 我懂你
- 我可以帮你
- 重要风险提醒
- 你没问，但要注意
- AI 智能分析
- 一键开启专业守护
- 严重违规
- 立即预约专家
- 我是 AI / 我是 GPT / 我是 DeepSeek / ChatGPT 帮你 / Claude 帮你

### 内部字段（任何用户可见位置出现即为 P0）

- `fallback_reason`
- `safety_gate_replaced`
- `no_source_matched`
- `out_of_scope`
- `low_confidence`
- `unknown` / `null` / `undefined`
- `__answer_run_v1__`
- `domain` / `intent` / `safety_gate` / `cross_domain` 等枚举字段名
- 任何 sidecar JSON 原文

### 禁用模式（不是单词，但同样禁止）

- 项目管理软件腔：`TODO`、`pending`、`in progress`、`status: ...`、`progress 30%`
- 占位符痕迹：`[TBD]`、`___`、`<insert here>`、明显的模板未填残留
- 系统改写痕迹：`我理解你的问题是 …`（除非过方向校验）
- 内部逻辑外泄：`触发了 X 规则`、`管线判定为 Y`
- 法律承诺：`一定通过`、`一定不通过`、`保证`、`绝对`
- 营销夸张：`专业守护`、`一键搞定`、`极速通关`

## 6. Matter Draft 文案边界

> Matter Draft 是 L2 的最薄一层（见 `TEBIQ_CONTEXT_PACK.md §3`）。当前文案边界以**克制**为最高优先。

### 用户端用词

- ✅ "保存这个事项"
- ✅ "我的事项"
- ✅ "回到这个答案"
- ✅ "继续处理：……"
- ✅ "还差一步：……"

### 不用的词（C 层）

- ❌ "行动包" — 项目管理腔
- ❌ "任务" / "task" — 项目管理腔
- ❌ "TODO" — 项目管理腔
- ❌ "进度 / progress / status pending" — 项目管理软件感
- ❌ "deadline" 单独使用（具体期限可以写"X 月 Y 日前"）
- ❌ "案件" — Matter Draft 不等于正式案件，不要让用户误以为开了正式 case

### 边界

- Matter Draft 是**回答后的轻保存**，不等于正式案件
- 不展示分配 / 负责人 / 状态机 / 审批流
- 不引导用户认为 TEBIQ 在"代办"案件 — TEBIQ 不是代理，专业代办由行政书士做

## 7. DOMAIN 与 VOICE 的关系

| 维度 | DOMAIN-CC | VOICE |
|------|-----------|-------|
| 角色性质 | 在留语义复核 / 行政书士助理型审查 | 用户端表达手艺人 |
| 输入 | Eval Lab 答案对（裸 DeepSeek vs TEBIQ） | Task Card + DOMAIN 标注 |
| 输出 | `severity` / `must_have` / `must_not_have` / `should_handoff` / `action` 等结构化标注 | 一段或一组用户可见文案 |
| 频次 | 持续（Eval 数据闭环主力） | 按需（仅在 Task Card 出现时） |
| 不做 | 直接落用户端文案 | 发明法律事实 / 决定页面结构 |

### 联动规则

- VOICE 输出的文案如**涉及专业事实**（手续名、期限、条件、政策依据），必须**已经有 DOMAIN-CC 标注的 `must_have` / `must_not_have` 作为约束**
- 如标注缺失，VOICE 必须停止，要求 DOMAIN-CC 先标注，**不允许 VOICE 自己补**
- DOMAIN-CC 不直接写用户端文案 — 标注的是语义边界，不是表达方式
- 任何文案如涉及专业事实但**找不到事实卡 / DOMAIN-CC 标注 / DeepSeek 原文支撑**，必须**等真人确认或事实卡入库**，不允许由 AI 任何窗口发明

## 8. 已观察的"黄金方向"参考片段（B 层）

> 这些片段不是冻结文案，是**风格 + 信息密度**的参考样本。在对应场景的 Eval 标注积累足够、产品负责人裁决前，仍然算 B 层。

**回答页样本（经管签搬办公室场景，B 层）：**

> 要通知，而且不只是改地址。  
> 公司登记和入管届出要分开处理。做了法务局的本店迁移登记，不代表入管那边也知道了。  
> 真正容易漏的不是届出本身，而是新办公室能不能撑住下次更新的审查。共享办公、住宅兼办公、短期租约，都会让下次经管更新变得更难解释。  
> 现在先做三件事：确认办公室类型、确认是否跨区搬迁、留好租赁合同 / 办公室照片 / 公司名牌 / 信箱证据。

**事项卡样本（B 层）：**

> 还差一步：把上次更新的资料找出来，作为这次的对比基础。

**handoff 样本（B 层）：**

> 这一步建议找行政書士确认，特别是：补材料期限、不许可通知书的回应、配偶离婚后的资格变更。

> 如涉及具体期限、手续名、适用条件，必须来自 DOMAIN-CC 标注或事实卡，不能由文案窗口发明。

## 10. 文案落地后回路

任何用户端文案落地后，必须：

1. 进入下一轮 Eval 抽样（与该场景对应的题目）
2. QA 做 Semantic QA 回归（见 `TEBIQ_QA_GATES.md` §3）
3. 如出现"答得没原 DeepSeek 好"的情况，文案必须回滚或重写

文案不是写完就结束 — **每段已上线文案都隐含一个未来可推翻它的 Eval 抽样契约**。
