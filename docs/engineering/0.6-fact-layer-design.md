## Goal

Wire the AI-first Fact Layer into production `/api/consultation/stream`:

```
user question
  → fact matcher (loads only state=human_approved cards)
  → matches → inject card.injection_format into system prompt
  → record fact_card_ids on the consultation row
  → Learning Console / /c/[id] / /me/consultations show injected cards
```

No card content authored by ENGINE. ENGINE is the runtime; content
lives in `docs/fact-cards/*.md` (filesystem) and `fact_cards` table
(runtime).

---

## State machine (gating)

| state | can match? | can inject? |
|---|---|---|
| `draft` | no | no |
| `ai_extracted` | yes (dry-run) | no |
| `ai_verified` | yes (dry-run) | no (PL override only) |
| `human_approved` | **yes** | **yes** (production) |
| `disabled` | no | no |
| `needs_review` | no | no |
| `conflict` | no | no |

Production loader **MUST** filter `state = 'human_approved'`. No bypass.

Internal dry-run endpoint `/api/internal/fact-layer/dry-run` may load
`ai_extracted` and `ai_verified` for testing. This endpoint is gated
by `EVAL_LAB_ENABLED` (existing pattern) and never reachable from
the user-facing app.

## Data model

### Filesystem (canonical source-of-truth)

`docs/fact-cards/<slug>.md` with YAML frontmatter (see existing
`keiei-kanri-2025-10.md` as the canonical shape).

Editing the file in repo + PR review = the human approval gate.
Merging the PR = approval. Loader reads filesystem at boot or via
hot-reload signal.

### Runtime table (Drizzle migration `0025_fact_cards.sql`)

```sql
CREATE TABLE fact_cards (
  fact_id           text PRIMARY KEY,
  title             text NOT NULL,
  state             text NOT NULL,        -- enum: draft|ai_extracted|ai_verified|human_approved|disabled|needs_review|conflict
  risk_level        text NOT NULL,        -- enum: P0|P1|P2
  confidence        text NOT NULL,        -- enum: low|medium|high
  applies_to        jsonb NOT NULL,       -- string[]
  trigger_keywords  jsonb NOT NULL,       -- string[]  (mirror of common_user_phrases keywords)
  injection_text    text NOT NULL,        -- the resolved injection_format block (TODAY_ISO substituted at request time)
  source_urls       jsonb NOT NULL,       -- string[]
  reviewer          text,
  last_verified_at  timestamptz NOT NULL,
  approved_at       timestamptz,
  approved_by       text,
  filesystem_path   text NOT NULL,        -- e.g. docs/fact-cards/keiei-kanri-2025-10.md
  content_hash      text NOT NULL,        -- sha256 of file content for drift detection
  created_at        timestamptz NOT NULL DEFAULT now(),
  updated_at        timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX fact_cards_state_idx ON fact_cards(state);
CREATE INDEX fact_cards_risk_idx ON fact_cards(risk_level);
```

### Consultation extension (Drizzle migration `0026_consultation_fact_cards.sql`)

```sql
ALTER TABLE ai_consultations
  ADD COLUMN fact_card_ids jsonb NOT NULL DEFAULT '[]'::jsonb;
-- jsonb array of fact_id strings injected into this consultation
```

## Sync pipeline (filesystem ↔ DB)

`scripts/fact-layer-sync.ts`:

1. Glob `docs/fact-cards/*.md` (excl. `README.md`).
2. Parse YAML frontmatter + body.
3. Compute `content_hash`.
4. For each card, upsert into `fact_cards` table by `fact_id`.
5. Hard-fail if a card has `state: human_approved` but is missing
   any required field.
6. Hard-fail if a card declares an `official_sources[].url` outside
   the source whitelist (`README.md` declares the whitelist).
7. Run on Vercel deploy (post-build hook) and via manual CLI.

Drift detection: at boot, the production loader compares each row's
`content_hash` to the on-disk file. Mismatch → log warning + emit
metric `fact_layer.drift_detected`. (No auto-resync in production —
mismatch is a deploy bug.)

## Matcher (Workstream C0+C6 core)

`lib/answer/fact-layer/matcher.ts`:

```typescript
type Match = {
  fact_id: string;
  trigger: string;       // which keyword/phrase fired
  score: number;         // 0..1
};

export function matchFactCards(question: string): Match[];
```

**v0.6 strategy** (intentionally simple, per PL "不追求完美"):

1. Normalize question: lowercase, strip punctuation, keep CJK.
2. For each card with `state = 'human_approved'`, scan
   `trigger_keywords` for substring presence.
3. Score = matched_keyword_count / total_keywords_in_card (capped at 1).
4. Filter: score >= 0.15 OR a P0 card matched any keyword (be liberal
   on P0 — false positive injection is far cheaper than missed P0).
5. Return top 3 matches by `(risk_level rank, score)`.

Future (post-0.6, NOT in scope now):

- Embedding similarity for `common_user_phrases`
- LLM-based intent → fact_id classifier
- Multi-card composition rules

**Cap**: maximum 2 cards injected per question (prompt budget). If
matcher returns >2, take top 2 by `(risk_level, score)`.

## Injection point

In `app/api/consultation/stream/route.ts`, **after** the system prompt
is assembled but **before** the user message is added:

```typescript
const matches = matchFactCards(question);
const factBlocks = matches.map(m => loadCard(m.fact_id).injection_text);
const todayIso = new Date().toISOString().slice(0,10);
const resolvedFactBlocks = factBlocks.map(b => b.replaceAll('{{TODAY_ISO}}', todayIso));

messages.push({
  role: 'system',
  content: resolvedFactBlocks.join('\n\n---\n\n')
});

// record injection
factCardIdsToPersist = matches.map(m => m.fact_id);
```

`completeAiConsultation` and `failAiConsultation` write
`fact_card_ids` to the consultation row.

**SSE event**: emit `fact_cards_injected` event after `received` and
before `first_token`, with `{ fact_card_ids: [...] }`. UI can use this
for the personalized risk hint (Workstream G).

## Workstream B integration (routing_status)

The `routing_status` SSE event (Workstream B) and `fact_cards_injected`
share the same keyword-bucket logic. Refactor proposal:

```typescript
// lib/answer/intent/keyword-buckets.ts
export const KEYWORD_BUCKETS = {
  keiei_kanri: { keywords: [...], status_label: '正在核对经营管理...' },
  nenkin_zeikin: { ... },
  gijinkoku: { ... },
  spouse_divorce: { ... },
};
```

Both `routing_status` (B) and `matcher` (C) consume `KEYWORD_BUCKETS`.
Single source of truth for keyword → bucket. Fact cards declare which
bucket(s) they participate in via `frontmatter.routing_buckets: []`.

- For each card's QA cases, hit prod `/api/consultation/stream`,
  collect answer, assert `must_have` strings present and `must_not_have`
  strings absent. Log `fact_card_ids` returned in the SSE.

## Deploy order

1. Migration 0025 (fact_cards table) — backwards compatible
2. Migration 0026 (ai_consultations.fact_card_ids) — backwards compatible (default `[]`)
3. Sync script + matcher + loader code
4. Stream route inject point — gated behind `FACT_LAYER_ENABLED=true` env (default false on first deploy)
5. Sync first card (`keiei-kanri-2025-10`) — but state=`ai_extracted`, will not inject
6. PL/DOMAIN approval PR flips state to `human_approved`
7. Flip `FACT_LAYER_ENABLED=true` after at least one human_approved card exists
8. Run QA regression suite

Rollback: flip `FACT_LAYER_ENABLED=false` (zero schema rollback needed).
