# founder_37 小野寺真｜行政手続 DX 创始人，前自治体电子申办顾问

## 1. 付费模式判断

**选择：选项 3。** 选 3。行政手续不是高频消费，限制搜索不自然。保存办理进度和期限提醒更符合日本行政 DX。转化 3% 已不错。

**¥980 的消费心理：** ¥980/月在行政 DX 里应解释为“个人手续台账”。如果只是 Q&A，太贵。

**我做过的订阅里，触发付费的最强信号：** 最强信号是用户把事项标为 进行中。TEBIQ 应让每个场景有 status，并在状态变更时触发付费。

## 2. 搜索作为主入口的判断

**判断：同意，但必须被场景化约束。** 自治体网站搜索可用率很低，用户靠场景导航。若做搜索，必须有 synonym dictionary。

**如果不是搜索，应该是什么：** 我支持搜索作为入口，但必须配合行政词同义词和场景推荐。

## 3. 多语言节奏判断

**判断：基本同意，但要分清 UI、源信息和协作者界面。** 中文后做日语 UI 有意义，因为日本官方 source 和 partner 都是日语。但 C 端仍需中文/英语/越南语解释。

**“日语 UI 给非日语用户用”是否可行：** 日语 UI 单独给非日语用户不够，但日语官方词层不可少。

## 4. 拍照定位降级判断

**判断：** 拍照降级合理，行政手续有很多没有文书的前置咨询。

**“图片识别 + OCR”作为单独产品的商业价值：** OCR 单独不是行政 DX。

## 5. 场景驱动 vs 知识库驱动判断

**哪种更容易撑订阅：** 场景驱动更能撑订阅，因为办理状态天然存在。

**我担心场景驱动的最大风险：** 风险是自治体差异。要在场景里区分 national rules 和 local confirmation。

## 如果我今天接手 TEBIQ，第一个月最重要的 1 件事

第一个月我会为每个场景加 responsible_office 和 local_variation 字段，并在 UI 里显示“需向所在自治体确认”。

## 创始人最该警惕的 1 件事

不要把日本行政做成全国统一流程。很多坑在自治体差异。
