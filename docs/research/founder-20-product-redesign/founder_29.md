# founder_29 彼得罗夫阿列克谢｜Microsoft Japan Productivity PM，企业知识与 Copilot 体验

## 1. 付费模式判断

**选择：选项 3。** 选 3。AI 搜索应该足够免费以证明价值；付费层应是 memory、workflow 和跨文档视图。企业 Copilot adoption 里，真正价值来自 context。

**¥980 的消费心理：** ¥980/月如果产品记得用户过去的文件，是合理的；如果只是更好的搜索，用户会拿它和 ChatGPT Plus、免费搜索比。

**我做过的订阅里，触发付费的最强信号：** 最强信号是跨文档引用：系统能说“你之前上传过同一机构文书”。TEBIQ 应在历史关联有价值时触发付费。

## 2. 搜索作为主入口的判断

**判断：同意，但必须被场景化约束。** 企业搜索如果没有 curated scope，使用很差。用户会问自然语言，但期待结构化答案。

**如果不是搜索，应该是什么：** 我支持自然语言框，但 retrieval 必须优先 curated scenarios 和 user archive。

## 3. 多语言节奏判断

**判断：需要修改节奏。** 中文第一。之后英语面向专业人士。日语 UI 应作为内部 ontology/source layer。

**“日语 UI 给非日语用户用”是否可行：** 日语-only UI 不是 accessibility。正确架构是日本语 ontology 上叠多语言标签。

## 4. 拍照定位降级判断

**判断：** 拍照降级正确，它只是一个 ingestion channel。

**“图片识别 + OCR”作为单独产品的商业价值：** OCR 是 commodity；archive-aware reasoning 才有防御性。

## 5. 场景驱动 vs 知识库驱动判断

**哪种更容易撑订阅：** 场景驱动更好，但必须有结构化数据；单纯知识库弱。

**我担心场景驱动的最大风险：** 风险是非结构化场景泛滥。没有 schema，30 个场景会变成 30 个手写页面。

## 如果我今天接手 TEBIQ，第一个月最重要的 1 件事

第一个月我会先定义 scenario schema：trigger、required dates、document types、offices、safe output boundary、reminder rules，再迁移 20 页。

## 创始人最该警惕的 1 件事

不要把场景页做成静态内容，要做 scenario engine。
