# founder_04 何雅琪｜大阪搬家/住居服务创业者，前生活服务 PM

## 1. 付费模式判断

**选择：选项 3。** 选 3。搬家类用户是脉冲需求，按搜索限制收费会被骂；但愿意为“搬家后 14 天内要办哪些手续”保存提醒付费。我的搬家工具付费转化约 4%，提醒短信打开率 62%。

**¥980 的消费心理：** ¥980/月对搬家用户像“搬家一次的附加服务”，可以接受；但如果过了搬家月还扣费，投诉会高。建议 ¥980/月保留，同时给 ¥1,980/90 天场景通行证。

**我做过的订阅里，触发付费的最强信号：** 最强信号是用户填写了搬家日期或在留期限。TEBIQ 应在场景页第一屏问日期，日期一填就出现“保存 30/90 天提醒”。

## 2. 搜索作为主入口的判断

**判断：不同意把裸搜索框替代首页 CTA。** 搬家服务里，用户搜索“転出届”这种词的比例低于 10%，更多人点“我要搬家”。日本手续词不是用户自然语言。

**如果不是搜索，应该是什么：** 不要裸搜索，做“生活事件入口”：搬家、结婚/离婚、生孩子、换工作、创业、父母来日。

## 3. 多语言节奏判断

**判断：需要修改节奏。** 中文优先。日语 UI 可作为官方词对照层，不应作为第二个完整 C 端语言。英语可以晚于越南语，因为越南/印尼/菲律宾用户手续痛点更强。

**“日语 UI 给非日语用户用”是否可行：** 非日语用户遇到行政词会截图去问朋友，日语 UI 无法降低焦虑。

## 4. 拍照定位降级判断

**判断：** 拍照降级没问题，但搬家场景中拍照很关键：市役所通知、水道、ガス、国保单都要进同一场景。

**“图片识别 + OCR”作为单独产品的商业价值：** OCR 单独能做一次性免费工具，不能做订阅。

## 5. 场景驱动 vs 知识库驱动判断

**哪种更容易撑订阅：** 场景驱动明显更能撑订阅，因为搬家有多部门、多日期、多文件。知识库只会让用户再去排清单。

**我担心场景驱动的最大风险：** 最大风险是场景页漏掉自治体差异。必须写“本页生成确认清单，不替代区役所公式案内”。

## 如果我今天接手 TEBIQ，第一个月最重要的 1 件事

第一个月我会上线“搬家后 14 天清单”这个场景作为样板，把市役所、入管、邮便、银行、ライフライン 分成 5 条任务，每条可上传一张文书。

## 创始人最该警惕的 1 件事

不要低估搬家场景。它不是签证核心，但频次和推荐性比续签高。
