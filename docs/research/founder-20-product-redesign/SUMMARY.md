## 1. 付费模式投票结果

- 选项 1 当前模式不变：4 票
- 选项 2 搜索次数限制：12 票
- 选项 3 浅功能全免费，深功能订阅：34 票

**主流结论：选项 3 明显胜出。** 50 位里多数认为 TEBIQ 的真实付费资产不是“再问一次”或“再拍一次”，而是用户把一件日本手续保存成进行中事项以后，系统能继续追踪期限、材料、历史和跨场景关系。

高质量异议：

- 选项 2 的支持者主要来自搜索、金融、学生/低收入群体、媒体和外资 AI 成本视角。他们担心免费搜索会带来 AI 成本失控，建议至少限制生成式搜索，而不是限制静态浏览。
- 选项 1 的支持者集中在 Trust & Safety、合规、Cookpad 订阅老兵。他们不是认为旧模式好，而是担心在产品信任还没稳时同时改入口和收费，会让用户更不安。
- 多位受访者认为 ¥980/月对“内容/搜索”太贵，对“手续工作台/历史档案/提醒”才合理。对于特定技能、留学、低收入群体，¥980 接近几顿饭；对于经管、外资白领、公司 HR，则低到需要更强信任包装。
- 关键数字判断：个人 C 端 option 3 的现实转化预估多落在 **3-6%**；低收入多语言群体可能只有 **1-2%**；高收入外籍专业人士可到 **5-8%**；B2B/机构付费可能比个人更健康。

## 2. 搜索主入口的支持度 + 关键反对意见

- 支持搜索主入口：19 票
- 反对裸搜索替代 CTA：31 票

结论不是“不要搜索”，而是 **不要做裸搜索框**。支持者也普遍要求搜索必须被场景库约束：autocomplete、真实问法、预制场景、query-to-scenario mapping、无结果兜底。

关键反对意见：

1. 日本手续用户不知道该搜什么。真实问题是“我现在遇到一件事”，不是“我知道官方关键词”。
2. 搜索框会把 TEBIQ 拉向 ChatGPT，而 TEBIQ 的差异化应是场景、档案、期限、历史。
3. 高风险问题会诱导用户问“我能不能”“会不会影响签证”，合规成本上升。
4. 弱势用户、学生、介护、特定技能用户更常拍照或点类别，不擅长搜索。
5. 搜索如果没有意图分类、同义词词典和 curated scenario，会很快暴露答案质量不稳。

最可执行的折中：**首页首屏做“我现在要办什么”自然语言输入 + 8-12 个高频场景 chips + 拍照快捷入口**。搜索负责捕捉 intent，场景页负责承接和付费。

## 3. 多语言节奏的真实可行性判断

- agree：7 票
- modify：37 票
- reject：6 票

共识：**“日语 UI 给非日语用户用”不是可靠的 C 端多语言策略。**

更细分的判断：

- 日语应该作为官方术语层、source layer、专家/HR/合伙人界面、分享给日本协作者的只读页，而不是给越南、菲律宾、印尼用户的主界面。
- 中文 1.0 继续打穿是对的。多位受访者建议先补繁体中文或英文高收入外籍版，再做越南语/印尼语/菲律宾语的合作试点。
- 越南语被多位一线创业者认为需求强，但必须绑定日本语学校、注册支援机构、雇主或社区渠道；没有母语审核和支持运营，不应独立上线。
- 英语的优势是高收入外籍专业人士和外资员工付费能力强，客服复杂度低于越南语，但痛点密度可能低于特定技能群体。

建议节奏：

1. 中文核心路径打穿。
2. 日文官方术语层 + 日语协作者/专家/HR share view。
3. 英文高付费场景小范围测试，或越南语特定技能场景与机构合作试点，二选一，不要同时做。
4. 每新增一种语言前，先准备母语免责声明、隐私同意、关键场景复核机制。

## 4. 拍照降级的风险共识

多数同意拍照从“唯一一级心智”降为“场景内输入工具”，但有明确底线：**拍照不能消失，也不能藏深。**

共识风险：

1. 当前线上版本最清楚的 activation 是“拍一份文书试试”。突然从首页拿掉，会损失短视频和新用户 demo 心智。
2. 弱势用户和非中文扩张用户更常用“拍照问人”，不一定会输入搜索。
3. OCR 单独商业价值低，但作为信任样例和获客 demo 很强。
4. 拍照结果必须自动进入场景/提醒/档案，否则只是一次性翻译，无法对抗 ChatGPT。

建议：保留首页浅层拍照入口，但主叙事从“拍照即懂”改为“这件事进入我的事项；有文书就拍进来”。

## 5. 场景驱动 vs 知识库驱动的分歧/共识

强共识：**场景驱动更能撑订阅，知识库更适合 SEO、信任证明和结果页支撑。**

原因：

- 场景天然有状态、期限、材料缺口、下一步，因此有订阅权益。
- 知识库让用户自己找，阅读完不一定形成资产。
- 场景页能承接真实咨询里的问题：办公室搬迁、带母亲、离婚、特定技能转换。

主要分歧：

- 搜索派认为场景应由搜索召回；流程派认为场景应由生活事件卡片选择。
- 合规派要求高风险场景只做事实归档和专家/官方窗口引导；增长派希望场景给更多明确下一步。
- 订阅派担心场景完成后用户立即取消，要求跨场景“我”视图和年度提醒。

最大风险：场景驱动一旦写得像“你应该这样申请”，就会越过合规边界；一旦写得像文章，又撑不起订阅。最佳形态是 **材料准备状态机**。

## 6. 第一个月该做什么的 Top 5 具体动作

1. **做 4 个真实咨询场景闭环，不先做 30 个薄页面。** 经管办公室搬迁、永住者带母亲、配偶签离婚、特定技能转换。每个场景固定结构：触发条件、3 个关键日期、6 个步骤、材料 slots、保存到提醒。

2. **搭建 query-to-scenario mapping。** 收集至少 100 个真实问法，映射到 20 个 scenario_key。搜索只负责意图捕捉，优先返回预制场景；未命中再保守 AI 解释。

3. **把 option 3 的付费墙放在“保存/提醒/30 天历史”时刻。** 免费看答案，付费保存进行中事项、邮件提醒、超 30 天历史、跨场景“我”视图。不要在用户还没拿到价值前限制搜索。

4. **先定义 scenario schema。** 每个场景必须有 trigger、visa_types、responsible_office、required_documents、deadline_rules、risk_boundary、save_fields、expert_handoff_condition。不要用静态文章硬堆。

5. **产品化隐私和数据保存。** 每次保存前明确：保存什么、不保存什么、保存多久、怎么删除、是否用于跨场景提醒。档案是护城河，也会成为最大的责任。

## 7. 创始人最容易低估的 3 个风险

1. **日语 UI 不是多语言过渡。** 它适合官方术语、专家端、HR 端、协作者页面，不适合作为越南/印尼/菲律宾/英语用户的主界面。

2. **搜索框会把产品拉向 ChatGPT 竞争。** 如果没有场景库、意图分类和档案记忆，用户会问：为什么不用 ChatGPT？TEBIQ 必须让搜索进入“我的事项”，而不是生成一段答案。

3. **场景越强，合规和个人信息风险越强。** 配偶离婚、经管办公室、特定技能转职、父母养老都不是普通内容题。必须有红线分类、数据权限和专家 handoff 条件。

## 一句话结论

TEBIQ 不应该从“拍照按钮”直接改成“搜索框”。更稳的迁移是：**首页从工具入口迁移到场景入口；搜索是意图捕捉器，拍照是证据输入器，付费点是进行中场景、提醒、历史档案和跨场景的我。**
