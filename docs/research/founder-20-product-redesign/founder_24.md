# founder_24 洛大卫｜Google Translate/Localization PM，多语言理解产品

## 1. 付费模式判断

**选择：选项 2。** 选 2。多语言 AI 成本随 query 增长很快，应限制免费搜索和拍照。翻译工具常见大量低意图免费使用，quota 能保护毛利。

**¥980 的消费心理：** ¥980 是高级翻译/手续工具价。用户会和免费 Google Translate、ChatGPT 比，必须靠领域记忆和历史档案解释价格。

**我做过的订阅里，触发付费的最强信号：** 最强信号是用户补充上下文或修正问题。TEBIQ 应记录 context_added，而不只是 query_count；愿意填签证类型和日期的人才是高意图。

## 2. 搜索作为主入口的判断

**判断：同意，但必须被场景化约束。** 翻译工具里输入框有效，因为用户会粘贴文本；但场景发现仍需要建议词。

**如果不是搜索，应该是什么：** 我支持搜索做主入口，但不要叫搜索，叫“输入你现在遇到的事”。它应同时接受文本、照片和文书片段。

## 3. 多语言节奏判断

**判断：拒绝这个节奏。** 我拒绝当前节奏。应是中文第一，然后英语和越南语的 native UX；日语不是桥，是 source language。

**“日语 UI 给非日语用户用”是否可行：** 日语 UI 给非日语用户是误解。正确做法是在母语解释中保留日语术语。

## 4. 拍照定位降级判断

**判断：** 拍照降级可以，照片只是输入模式之一；用户很多时候在有文书前先描述场景。

**“图片识别 + OCR”作为单独产品的商业价值：** OCR 单独几乎无防御性。领域抽取 + 用户记忆才有防御性。

## 5. 场景驱动 vs 知识库驱动判断

**哪种更容易撑订阅：** 场景驱动更能订阅，但搜索需要覆盖长尾表达。

**我担心场景驱动的最大风险：** 多语言最大风险是翻译后的场景名对不上日本法律概念。底层必须维护日本语 canonical ontology。

## 如果我今天接手 TEBIQ，第一个月最重要的 1 件事

第一个月我会先建日本行政术语 canonical ontology，中文标签跑通后，再让英语/越南语标签映射同一个 scenario_id。

## 创始人最该警惕的 1 件事

本地化不是翻译屏幕，是把用户语言映射到日本行政概念。
