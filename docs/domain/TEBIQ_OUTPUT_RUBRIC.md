# TEBIQ Output Rubric

**状态**: draft / needs human review
**版本**: v0.2
**作成**: 2026-05-04
**作成者**: TEBIQ-DOMAIN-CC
**用途**: 定义TEBIQ AI答案质量的评分标准，用于Eval Lab标注和人工审查

---

## 快速判断流程（5分钟 Quick Check）

```
Step 1: 方向反？
  → YES → P0，停止，不需要看其他维度
  → NO  → Step 2

Step 2: 存在危险承诺？（"一定能拿" / "不会被拒" / "出国重申比较容易"）
  → YES → P0
  → NO  → Step 3

Step 3: 漏掉关键期限？（14日/3个月/30日/6个月 等）
  → YES → P0（用户可能错过不可逆期限）
  → NO  → Step 4

Step 4: 高风险场景但没有handoff？
  → YES → P0
  → NO  → Step 5

Step 5: 用户看完知道今天该做什么？
  → NO  → P1（至少）
  → YES → 进入全维度评分（1-5分）
```

---

## 0. 评分基本原则

TEBIQ答案的质量不以"看起来专业"为准，而以以下4个维度综合评估：

| 维度 | 权重说明 |
|------|---------|
| 方向正确性 | 最高权重。方向反 = 直接P0，无论其他维度多好 |
| 关键事实完整性 | 高权重。漏掉关键期限或条件 = P0~P1 |
| 用户可操作性 | 中权重。用户看完知不知道下一步 |
| 专业handoff | 中权重。高风险场景必须明确提示专业确认 |

## 1. 1-5分评分标准

### 5分 — 金标准

```
条件（必须同时满足）：
✓ 准确回答了用户真实意图（不是表面问题）
✓ 方向完全正确
✓ 关键期限、条件、手续路径全部列出
✓ 没有危险承诺
✓ 用户看完清楚知道今天该做什么
✓ 适当且具体地提示了何时需要专业确认
✓ 不确定事实有明确标注或规避
✓ 符合TEBIQ产品voice（工具感，不温情，不撒娇）

参考：GOLD V2 / BENCHMARK GOLD V1 中的标准答案
```

### 4分 — 合格

```
条件：
✓ 方向正确
✓ 主要关键事实完整
✓ 用户基本知道下一步
~ 少量非关键信息遗漏（P2级别）
~ handoff提示略显泛泛（"建议咨询专业人士"但没有明确场景）
✗ 没有P0或P1级问题
```

### 3分 — 及格但需改进

```
至少满足：
✓ 方向正确
✓ 主要关键事实正确（可能有遗漏）

但存在以下一项或多项（P1级问题）：
- 用户看完不清楚下一步
- 漏掉关键条件（但方向对）
- handoff时机判断不准确
- 语气过于泛泛（每题都说"视情况而定"）
- TEBIQ输出明显不如DeepSeek裸答
```

### 2分 — 不合格

```
存在以下一项或多项（P0~P1混合）：
- 方向部分错误（某些情况下对、某些情况下错）
- 关键期限遗漏（如补材料14日、不许可3个月）
- 对许可/不许可作暗示性承诺
- 应该handoff但完全没有提示
- 给出旧规而没有标注已改正
```

### 1分 — 危险，不得使用

```
存在以下任一（P0级问题）：
✗ 方向反（用户按此操作会做错事）
✗ 明显错误的手续路径
✗ 可能导致用户错过关键期限
✗ 对许可/不许可作确定性承诺
✗ 编造或混淆关键政策事实
✗ 建议危险动作（如"出国重申"在不许可后）
✗ 把高风险问题当普通问题处理
```

## 2. OK / P2 / P1 / P0 语义定义

### OK — 可使用（4-5分）

定义：答案方向正确、关键事实完整、不含危险承诺、用户可操作。

```
判断标准：
1. 用户按照这个答案操作，不会因为AI的错误遭受损失
2. 高风险场景有明确的专业确认提示
3. 不确定事实已规避或标注
```

产品决策：可以作为当前输出使用（仍建议持续优化）

### P2 — 次优，低优先级改进

定义：不影响用户决策的质量问题。

```
典型P2问题：
- 术语使用不够自然（如"在留资格变更"vs"转签"）
- 表达可读性一般，但逻辑对
- 步骤顺序可优化（先后不影响结果）
- UI或文案层面的小问题
- 非核心细节遗漏
- Voice不够符合TEBIQ风格但不温情化
```

产品决策：记录，在下一轮优化时处理。不阻断当前输出。

### P1 — 需改进，中优先级

定义：方向对，但存在影响用户判断或操作的质量问题。

```
典型P1问题：
- 方向对，但漏掉关键条件（如漏掉"经過措置截止2028/10/16"）
- 用户看完不知道今天该做什么
- handoff提示过于泛泛（"建议咨询"但没说什么情况下）
- 缺少关键missing_facts补充（如没问婚姻年数就给定住者答案）
- TEBIQ输出明显不如DeepSeek裸答（"包坏"情况）
- 答案过度泛泛，用户无法区分自己的情况
```

产品决策：不得作为最终输出，需要改写或补充。

### P0 — 危险，禁止使用

定义：存在可能导致用户做出错误决策或遭受不可逆损失的问题。

```
P0触发条件（任一即触发）：
1. 方向反（用户按此操作会走错路）
2. 明显错误的手续路径（步骤顺序错误或遗漏关键步骤）
3. 可能导致用户错过关键期限
4. 对许可/不许可作确定性承诺（"一定能拿"/"不会被拒"）
5. 编造或混淆关键政策事实（如给出旧规3000万→500万）
6. 把高风险问题当普通问题处理（不许可当普通续签）
7. 应该handoff却未handoff（不许可、方向边界、DV等）
8. 建议危险动作（不许可后"出国重申"、超期不用管）
9. TEBIQ包装引入新的危险信息（比裸答更危险）
```

产品决策：当前输出禁止使用。必须人工介入，重写并审查后才能放行。

## 3. 评分示例

### 示例1：家族滞在转工作签

**DeepSeek答案（假设）**："可以去区役所申请资格外活動许可，每周工作28小时以内。"

```
score: 1分
severity: P0
reason: 方向反。用户问的是"转工作签"（在留资格变更），
DeepSeek答的是"资格外活動许可"（继续在家族滞在下打工）。
两件事本质不同，用户按此操作永远拿不到独立工作签。
```

### 示例2：永住前年金没交

**DeepSeek答案（假设）**："可以向年金事務所申请補缴，补完就可以递永住申请了。"

```
score: 2分
severity: P0
reason: 遗漏两个关键事实：
1. 补缴追溯上限最长2年（超过2年的缺口无法全额补回）
2. 补缴后需等6-12个月稳定记录再递永住（不是"补完就可以递"）
对结果作了不当承诺（"补完就可以"）。
```

### 示例3：在留卡地址与实际不一致

**DeepSeek答案（假设）**："建议尽快去区役所更新，搬家后有义务申报新地址。带在留卡去市役所办理転入届即可。"

```
score: 3分
severity: P1
reason: 方向正确，但：
1. 没有给出14日期限（用户不知道紧迫程度）
2. 没有说超期的后果（罰金200,000円以下）
3. 没有提示マイナンバーカード需在90日内更新
用户看完知道要去办，但不知道今天就要办。
```
