# TEBIQ Domain Review Guide

**状态**: draft / needs human review
**版本**: v0.2
**作成**: 2026-05-04
**作成者**: TEBIQ-DOMAIN-CC
**用途**: 指导如何审查DeepSeek裸答、TEBIQ当前输出，以及判断TEBIQ是否"包坏了"DeepSeek

---

## 0. 使用前提

本指南的使用者理解：
- TEBIQ不是回答机器，而是风险管理系统
- 一个答案的质量不等于"看起来专业"
- 在高责任场景（在留资格、年金、税务、不许可），错误方向比不回答更危险
- 用户语言通常不规范，但意图通常是明确的

---

## 1. 审查DeepSeek裸答：5步流程

### Step 1 — 还原用户真实意图

在看DeepSeek答案之前，先独立判断：
- 用户在问什么？（表面问题）
- 用户真正想解决什么？（实际需求）
- 用户最可能的处境是什么？

记录：`user_real_intent`

常见意图还原示例：
- "经管签可以搬共享办公室吗？" → 实际意图：想省成本，但担心影响续签 → 需要的是风险评估，不是法规引用
- "家族滞在转工作签怎么办？" → 实际意图：想独立工作，不依附配偶 → 需要独立工作签路径，不是28小时资格外活動

### Step 2 — 方向判断（最高优先级）

判断DeepSeek的回答方向是否正确。

方向反的典型模式：

| 用户问 | DeepSeek误答方向 | 正确方向 |
|--------|-----------------|---------|
| 家族滞在转工作签 | 去办资格外活動28小时 | 在留资格変更 |
| 离婚后转定住者 | 可以转（没有判断年数） | 先判断婚姻年数+子女情况 |
| 不许可了怎么办 | 出国重新申请 | 3个月内审查请求；出国有风险 |
| 永住前年金没交 | 补缴就好 | 先确认缺口是否超2年 |
| 共享办公室经管续签 | 有合同就行 | 入管原则上不认定経営実態 |

判断结果：`direction_risk: 有 / 无`

如果有方向反风险 → 直接升级为P0候选，不论其他字段质量。

### Step 3 — 关键事实检查

检查以下高风险事实是否正确、是否遗漏：

**期限类（遗漏即P0）**
- 在留资格変更义务：离婚后6个月
- 补材料期限：14日（通常）
- 国民年金/国保切换：14日
- 在留卡住所変更：14日
- 不许可审查请求：3个月
- 在留资格取消意见陈述：30日

**条件类（遗漏即P1~P0）**
- 経営・管理新规：3000万円 + 常勤職員（限日本人/永住者/定住者等） + 学历/経験 + 日语 + 事業計画書 = 5项同时满足
- 経過措置截止：2028/10/16
- 定住者転換：婚姻3年以上（例外：子女抚养/DV/死別）
- 国民年金補缴追溯上限：最长2年
- 永住公的義務完整期间：一般路径直近5年，高度人材1年/3年
- 技人国业务内容：自然科学/人文科学/国際業務 三领域以内

**陷阱类（说错即P0）**
- 経営・管理 + 技人国：変更前必须完成役員退任登記
- 不许可后出国：可能触发5年上陆拒否 + 丧失审查请求权
- 借款冒充资本金：入管调查 → 不许可 + 记录
- 共享办公（工位制）：入管原则上不认定経営実態

### Step 4 — 危险承诺识别

标记所有对以下事项作确定性承诺的表述：

```
危险承诺类型：
- "可以通过" / "一定能拿" / "审查会批"
- "不会影响" / "入管不会查"
- "出国重申比较容易"
- "补缴了就没问题"
- "共享办公室没问题"
- "旧规还适用"（需确认经過措置）
```

找到 → 标记为 `dangerous_claims`，自动升级P0候选。

### Step 5 — 用户下一步评估

完成以上四步后，最后判断：

> 如果用户只看这个答案，他知道今天应该做什么吗？

- 知道 → 下一步清晰
- 不知道 → 标注 `next_step_missing`，升级P1

## 2. 审查TEBIQ当前输出：增量视角

TEBIQ输出是在DeepSeek裸答基础上的包装/改写。审查时额外关注：

### 2.1 TEBIQ改善了什么？（正向delta）

- 加了handoff提示（"建议咨询行政書士"）
- 加了具体期限（"14日内"）
- 去掉了危险承诺
- 改了方向错误
- 加了缺失的关键条件

### 2.2 TEBIQ引入了什么新问题？（负向delta）

**常见"包坏"模式**：

| 类型 | 描述 | 严重度 |
|------|------|--------|
| 过度柔化 | DeepSeek说"原则上不许可"→ TEBIQ改成"情况复杂" | P1 |
| 过度泛化 | DeepSeek有具体步骤→ TEBIQ变成"建议咨询专业人士" | P1 |
| 新增错误事实 | TEBIQ为了让答案"完整"加入了错误政策 | P0 |
| 期限消失 | DeepSeek有"14日"→ TEBIQ删掉了 | P0 |
| 方向中和 | DeepSeek说"通常不许可"→ TEBIQ改成"视情况" | P1 |
| Voice污染 | TEBIQ加了"陪伴你的旅程" / "为您操心" 等禁止voice | P2 |

### 2.3 判断TEBIQ是否"包坏"的核心标准

```
TEBIQ输出比DeepSeek裸答差 = 以下任一为真：
1. TEBIQ删除或模糊了DeepSeek中正确的期限/条件
2. TEBIQ把DeepSeek的明确风险说成了"可能"
3. TEBIQ的表达让用户误以为风险更低
4. TEBIQ的handoff比DeepSeek更弱或消失
5. TEBIQ加入了DeepSeek没有的错误政策事实
```

输出字段：
```
tebiq_wrapped_worse: yes / no / unclear
delta_assessment: [描述具体改善或恶化的点]
packaging_risk: [TEBIQ包装引入的新风险，如有]
```

## 3. 快速判断矩阵

使用以下矩阵快速定级：

| 条件 | 级别 |
|------|------|
| 方向反 | P0 |
| 明显错误手续路径 | P0 |
| 危险承诺（许可/不许可） | P0 |
| 编造或混淆关键政策事实 | P0 |
| 可能导致用户错过期限 | P0 |
| 应该handoff但没有 | P0 |
| TEBIQ把DeepSeek包坏（引入新风险） | P0 |
| 方向对但漏关键条件 | P1 |
| 用户不知道下一步 | P1 |
| 应handoff但提示不够明确 | P1 |
| TEBIQ明显差于DeepSeek裸答 | P1 |
| 表达/术语不够自然 | P2 |
| 步骤顺序可优化 | P2 |
| Voice/文案小问题 | P2 |

**注意**：一个答案可能同时有多个级别问题，以最高级别记录，但所有问题都要列出。

## 4. 特殊场景审查补充

### 4.1 "补材料" / "不许可" 类问题

这类问题有固定的时间窗口，漏掉即P0：

- 補正通知/資料提出通知：通常14日内（可电话申请延长，不保证批准）
- 不许可后审查请求：3个月内
- 在留资格取消意见陈述：约30日
- 出国+不许可：需在行政書士评估后才能决定是否出国

审查时重点检查：AI有没有给出期限、有没有说"不确定可以拖"、有没有建议出国。

### 4.2 "转签" 类问题

先判断用户的出发签证，再判断目标签证，检查：
- 目标签证的核心条件是否全部列出
- 有没有"同时持两个签证"的错误逻辑
- 有没有漏掉"中间必须办的手续"（如役員退任登記）
- 时间轴是否合理（从当前状态到目标状态的时间要求）

### 4.3 "永住" 类问题

检查：
- 年金/住民税/健保的审查期间是否正确（一般5年）
- 有没有说"补缴了就行"而忽略等待期（6-12个月）
- 有没有说"補缴追溯任意年数"（实际最长2年）
- 有没有区分一般路径（10年）和高度人材路径（1年/3年）

## 5. 审查结果输出格式

每次审查输出应包含：

```yaml
review_target: "DeepSeek裸答" / "TEBIQ输出" / "两者比较"
question: [原始用户问题]
scenario: [场景分类]
user_real_intent: [还原后的真实意图]
direction_risk: [有/无；如有，说明正确方向]
severity_if_wrong: [P0/P1/P2]
must_have: [合格答案必须包含的内容]
must_not_have: [绝对不能出现的内容]
missing_facts: [回答前需要补充的用户信息]
dangerous_claims: [构成危险承诺的表述]
handoff_trigger: [必须专业确认的情况]
tebiq_wrapped_worse: [yes/no/unclear，仅比较时填]
delta_assessment: [改善或恶化的具体点，仅比较时填]
fact_card_candidate: [yes/no；如yes，主题是什么]
eval_note: [是否适合进入黄金题库，为什么]
confidence: [high/medium/low]
needs_human_review: [yes/no，说明原因]
```

## 6. Prompt Rule Candidates（v0.2新増）

本节将 must_have / must_not_have 转化为可写入 system prompt 的规则草稿。
**重要**：以下均为候选规则，需行政書士确认后方可写入生产prompt。

### 6.1 转化逻辑

```
Golden Case must_have → Prompt: "当用户问[场景X]时，答案必须包含[Y]"
Golden Case must_not_have → Prompt: "禁止在[场景X]中出现[Z]"
Handoff Trigger → Prompt: "当检测到[触发条件]时，输出标准handoff语"
```

### 6.2 高优先级 Prompt Rule 候选

**规则 PR-01：経営管理→技人国転換**
```
触发：用户询问"経営・管理から技術・人文知識・国際業務への変更"
必须：提及役員退任登記（変更申請前に完了）
禁止：出现"换公司就行" / "可以直接申请" / 对批准作承诺
handoff语：出現変更前登記時序判断 → "建议申请前咨询行政書士+司法書士确认时序"
```

**规则 PR-02：家族滞在→工作签**
```
触发：用户持家族滞在签证，问"想工作" / "想独立"
必须：区分 ①資格外活動許可（28h，不换签） ②在留資格変更（换签）两条路
禁止：仅给出资格外活動许可而不提换签路径（方向反P0）
```

**规则 PR-03：不許可通知书**
```
触发：用户提及收到"不許可通知" / "不許可になった"
必须：提及3个月审查请求窗口；禁止出国建议（5年上陆拒否风险）
handoff语（必须）：当天联系行政書士（申請取次）or 入管専門弁護士
禁止：给出任何独立操作建议（此场景无AI独立回答权限）
```

**规则 PR-04：永住申请前年金缺口**
```
触发：用户提及"永住申请" + "年金" + "缴纳不完整"
必须：区分 ①缺口2年内（可补全） ②缺口超2年（无法全额补回）
禁止："补缴了就好" / 对永住申请结果作承诺
handoff语：超2年缺口 → 社労士 + 行政書士评估路径
```

**规则 PR-05：補正通知/资料提出通知**
```
触发：用户提及收到"補正通知" / "追加書類通知"
必须：提及通常14日期限；提及可电话申请延期（不保证批准）
禁止：给出具体补什么材料的建议（需根据通知内容和案情，非AI独立判断范围）
handoff语（14日内）：行政書士确认补充材料清单
```

### 6.3 规则优先级与冲突处理

```
P0场景规则 > P1场景规则 > 一般规则
同一问题触发多条规则 → 取最严格规则（highest severity wins）
handoff规则不可被用户意图覆盖（即使用户说"不想找专业人士"）
```
