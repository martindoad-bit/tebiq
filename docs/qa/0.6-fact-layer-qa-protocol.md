```text
你是 TEBIQ QA。本次任务：把每张 fact card 自带的 qa_cases
转成对生产 /api/consultation/stream 的回归测试，并产出 pass/fail 报告。

主任务（按 docs/qa/0.6-fact-layer-qa-protocol.md）：
1. 接收 GM 派发的 1+ 张 fact card（已 merged）
2. 从每张卡 §qa_cases 抽题
3. 对生产端发起 SSE 请求
4. 用 must_have / must_not_have 对答案做断言
5. 记录 first_token_ms, total_latency_ms, fact_card_ids hit
6. 产出 pass/fail 报告 issue 给 GM

绝对边界：
- 不改 fact card 内容（DOMAIN 才能改）
- 不改 ENGINE / matcher（GM 才能改）
- 不在 production 跑大批量测试（节制：每张卡 ≤5 题）
- 不重测 DOMAIN 已 reject 的卡

每张卡测完，按 §5 报告模板写 issue。
```

## 1. QA 在 0.6 中的角色

### DO

- 把 fact card 内置的 `qa_cases` 当回归题，跑生产 `/api/consultation/stream`
- 验证 must_have 字符串出现、must_not_have 字符串不出现
- 验证 SSE 中 `fact_cards_injected` 事件命中预期 fact_id（matcher 验证）
- 记录速度指标（first_token_ms, total_latency_ms, output_chars）
- 跨多张卡跑同一题，验证多卡注入 / 互不干扰
- 周回归（GM 排期）

### DON'T

- ❌ 改卡内容（DOMAIN 才能改）
- ❌ 改 matcher / injection 逻辑（GM/ENGINE）
- ❌ 在生产端跑性能压测（影响真用户）
- ❌ 重测 DOMAIN 已 reject 或 disabled 的卡
- ❌ 把 QA 不通过当作"模型差"判断 — 先排查是 prompt / matcher / fact 命中哪一层

## 2. 测试输入来源

每张 fact card 的 markdown 文件包含 `## qa_cases` section。
QA 提取格式：

```yaml
question: <string>
must_have: [string list]
must_not_have: [string list]
```

注意 must_have / must_not_have 是**关键短语**，不是逐字精确匹配。可以用 substring（normalized for whitespace + simplified Chinese variants）。

---

## 3. 测试方法

### 3.1 SSE 调用

```bash
curl -fsS -N --max-time 180 \
  -H 'Content-Type: application/json' \
  -d '{"question":"<the question>"}' \
  https://tebiq.jp/api/consultation/stream
```

收集事件：

- `received` → consultation_id
- `fact_cards_injected` → 命中的 fact_id 列表 + decision (`inject` / `hint_only` / `drop`)
- `first_token` → first_token_latency_ms
- `answer_chunk` → 累积全文
- `completed` / `partial` / `failed` / `timeout` → 终态

### 3.2 断言

| 断言 | 通过条件 |
|---|---|
| matcher 命中 | `fact_cards_injected.items` 含期望 fact_id |
| matcher decision | decision = `inject`（如 ai_verified high/critical 应注入）|
| must_have | 全部短语在累积答案中出现（normalized substring） |
| must_not_have | 全部短语**不**出现 |
| status | `completed` 或 `partial`（非 `failed` / `timeout`） |
| speed (smoke) | first_token < 30s, total < 120s（Pro thinking baseline 4-25s 是常态） |

### 3.3 Test harness 建议

QA 可建一个简单 Node/TS 脚本（不进 production，仅 internal）：

```typescript
// scripts/fact-layer-qa-runner.ts (示例)
async function runCard(card: FactCard) {
  for (const qa of card.qa_cases) {
    const events = await streamConsultation(qa.question);
    const answer = collectAnswer(events);
    const matches = collectFactCardsInjected(events);
    
    const matcherOk = matches.includes(card.fact_id);
    const mustHaveOk = qa.must_have.every(s => answer.includes(s));
    const mustNotHaveOk = qa.must_not_have.every(s => !answer.includes(s));
    
    yield { question: qa.question, matcherOk, mustHaveOk, mustNotHaveOk, ... };
  }
}
```

可以 commit 到 `scripts/qa/` 后续复用，但本轮**不要**部署成自动化 CI gate（避免 noise）。

### 3.4 限速

- 同卡 qa_cases ≤ 5 题
- 跨卡总测试 ≤ 30 题 / 周
- 不要在用户高峰期跑（建议 JST 03:00-06:00 跑回归）

## 4. 何时跑

| 触发 | 范围 |
|---|---|
| 新 fact card merged | 该卡所有 qa_cases |
| matcher 代码改动 | 当前所有 ai_verified/human_reviewed 卡 quick smoke (1 case/card) |
| `FACT_LAYER_ENABLED` 翻 true | 全部 ai_verified/human_reviewed 卡完整回归 |
| Production model 切换 | 全部 ai_verified/human_reviewed 卡完整回归 |
| Incident review 后 | 涉事卡 + 同 keyword bucket 邻近卡 |
| 周回归 | GM 排期，按主题轮换 |

## 6. 失败的归因路径

QA 不直接修，但 root cause hypothesis 要写：

| 失败模式 | 可能层 | 归还谁 |
|---|---|---|
| matcher 没命中预期 fact_id | matcher (lib/answer/fact-layer/matcher.ts) | GM/ENGINE |
| matcher 命中但 decision != inject | gate logic | GM/ENGINE |
| inject 了但 must_have 不出现 | injection_certain_block 缺关键事实 | DOMAIN audit |
| inject 了但 must_not_have 出现 | model 自由发挥违反约束 → prompt 强化 | GM (调 voice) |
| status = failed/timeout | stream 链路 | GM/ENGINE |
| first_token > 30s | model / network | 监控但不一定 fix |
| 同一题不同 run 结果不一致 | model temperature / 非确定性 | 接受 baseline 噪声 |

## 7. 与其他角色的接口

| 来自 | 内容 | QA 怎么消费 |
|---|---|---|
| FACT (via GM PR) | qa_cases | 直接读 markdown |
| GM | 派发 PR 到 QA | 接 issue 跑测 |
| DOMAIN | audit verdict | QA 不重测 DOMAIN reject 的卡 |
| ENGINE | matcher / injection 改动 | QA 触发回归 |

QA 不直接对话 FACT / DOMAIN / ENGINE。所有沟通走 GM。
