**Truncation observations:**
- 3/5 questions ended mid-sentence (C-Q1, C-Q4, C-Q5) with `status: completed`
- The `completed` SSE event fires at token-budget exhaustion, not at semantic completion
- Users see an incomplete answer rendered as "完整回答" — no UI indicator of cut-off

**P2 finding:** Token budget truncation renders with `completed` status. Users have no indication the answer is incomplete. This is a product experience issue — see P2-B.

## Section F — Boundary / Abuse Tests

| Test | Expected | Got | Result |
|------|----------|-----|--------|
| F-1: follow-up, nonexistent parent_consultation_id | 404 parent_not_found | HTTP 404 `{"error":"parent_not_found","detail":"..."}` | ✅ |
| F-2: GET /c/nonexistent-id | 404 | HTTP 404 (Next.js not-found page) | ✅ |
| F-3: POST /api/consultation/stream with `{}` | 400 | HTTP 400 `{"error":"bad_request"}` | ✅ |
| F-4: follow-up, no viewer cookie (cookieless request) | 200 (parent also anonymous) | HTTP 200 — no mismatch (both cookieless) | ✅ |
| F-5: follow-up, parent 25h+ old (from WS-A 2026-05-07 07:19Z) | 410 parent_too_old | HTTP 200 — NOT 410 | ⚠️ P2 |

**F-5 analysis:** `PARENT_MAX_AGE_HOURS=24` defined in `lib/answer/followup/constants.ts` but
the 410 did not fire for a 25h+ old parent. The follow-up was accepted and completed
normally. Either the age check has not been deployed to production, or the constant is not
wired to the route handler. See P2-C.

**F-4 note:** `ENFORCE_VIEWER_MATCH=true` — viewer_mismatch 403 correctly only fires when
a viewer cookie exists on the parent AND the current request carries a different cookie.
Cookieless Python requests produce no viewer cookie on either side, so no mismatch.
Real user scenario (browser session → different browser session) would trigger 403 as
designed.

**Section F Verdict: PASS (F-5 logged as P2)**
